# 洪水预警系统主动预警与主动请示需求文档

## 1. 文档目的

本文档用于固化当前系统中“主动预警 + 主动请示”能力的实现边界与交互规则，作为后续继续演进、测试验收与文档同步的依据。

与早期版本相比，当前系统已经从“固定三类区域动作 + 规则草稿”升级为：

- 风险判定保持规则确定性
- 用户可见的问答、建议、区域动作、请示草稿、执行材料由 LLM 主导生成
- 区域动作允许开放式 `action_type`
- 前端通过全局对话框处理跨事件待确认区域 proposal

## 2. 当前目标

系统当前围绕以下目标工作：

- 以洪水模拟结果更新事件作为区域主动链路的正式入口
- 在达到高风险条件时主动生成区域级待确认动作
- 将真正的关键动作收口到 `commander` 的统一确认门
- 在不改变风险确定性判断的前提下，尽可能让建议与决策文案由模型完成
- 通过展示字段把开放式动作呈现为“指挥动作能力”，而不是原始技术 key

## 3. 术语定义

### 3.1 区域综合风险

指系统基于洪水模拟结果中的区域水深、水流与阈值，结合综合规则计算出的区域级风险结果。该结果不是由单个点位直接决定，而是由区域整体结果决定。

### 3.2 高风险阶段

当前实现中，`Orange` 和 `Red` 视为进入主动请示的高风险阶段。

### 3.3 新的高风险阶段

以下任一情况视为“新的高风险阶段”：

- 区域从非高风险重新进入 `Orange/Red`
- 区域在同一次高风险过程中从 `Orange` 升级为 `Red`

### 3.4 区域级 proposal

指面向整个区域生成的待确认行动项。每条 proposal 对应一条区域动作，由 `commander` 在统一确认对话框中逐条确认、驳回或编辑草稿后确认。

### 3.5 对象级追问

指用户围绕学校、社区、养老机构、居民点等具体对象继续发起问答。对象级追问当前只输出建议和解释，不默认生成新的可确认 proposal。

## 4. 风险分级与触发规则

### 4.1 风险分级

系统沿用现有四色风险分级：

- `Blue`
- `Yellow`
- `Orange`
- `Red`

### 4.2 触发线

- `Blue/Yellow`
  - 仅用于展示、记录和后台分析
  - 不进入区域主动请示
- `Orange/Red`
  - 进入高风险阶段
  - 允许区域 proposal 主动生成

### 4.3 正式触发源

主动请示的正式入口为：

- `POST /platform/events/{event_id}/simulation-updates`

系统不再依赖纯定时扫描“猜测”是否应该弹出请示，而是以模拟结果更新事件作为主起点。

## 5. 当前实现链路

### 5.1 区域主动链路

1. 接收洪水模拟结果更新事件
2. 根据水深、水流和阈值计算区域综合风险
3. 若进入 `Orange/Red`，发布 `simulation_updated` trigger
4. `PlanningAgent` 调用 LLM 生成区域动作集合
5. `RegionalProposalManager` 为每条动作生成或替换区域 proposal
6. SSE 推送全局 pending queue
7. 前端全局对话框立即打断当前界面
8. `commander` 逐条确认、驳回或编辑同类动作参数
9. 批准后由 LLM 生成执行材料与执行摘要，并落审计与执行日志

### 5.2 对象级链路

1. 用户发起对象级问答或 advisory 请求
2. 平台获取 impact、知识证据、上下文
3. LLM 生成中文建议、影响解释和 grounding 摘要
4. 当前默认不进入区域级可确认 proposal 队列

## 6. 区域 proposal 生成规则

### 6.1 生成主体

区域动作集合由 LLM 在 `regional_decision` 阶段生成，请示草稿由 `proposal_draft` 阶段生成。

### 6.2 动作类型

当前系统不再限制区域动作必须是固定三类。`action_type` 允许开放式值。

不过执行闭环仍通过 `execution_mode` 归一化为以下四类：

- `notification`
- `evacuation_task`
- `resource_dispatch`
- `generic_task`

说明：

- 已知动作可以走专门语义
- 未知动作统一走 `generic_task`
- 前端展示优先依赖展示字段，而不是直接显示 `action_type`

### 6.3 展示元数据

每条区域 proposal 当前应带以下展示字段：

- `action_display_name`
- `action_display_tagline`
- `action_display_category`

它们由模型在 `proposal_draft` 阶段一并生成，用于把开放式动作展示为“指挥动作能力卡”。

## 7. proposal 生命周期

### 7.1 状态

当前区域 proposal 支持以下状态：

- `pending`
- `approved`
- `rejected`
- `withdrawn`
- `superseded`

### 7.2 新阶段标识

系统通过 `risk_stage_key` 标识当前高风险阶段，生成规则保持如下：

- 非高风险进入 `Orange`：新建 `risk_stage_key`
- `Orange -> Red`：新建 `risk_stage_key`
- 同一级高风险持续：沿用当前 `risk_stage_key`
- 高风险解除后再次进入：重新生成 `risk_stage_key`

### 7.3 明显变化判定

proposal 是否发生明显变化，当前仍只基于以下三类内容生成 `system_version_hash`：

- `action_type`
- `high_risk_object_ids`
- `action_scope`

### 7.4 替换规则

- 若 `risk_stage_key` 变化：
  - 允许重新生成 proposal
  - 立即重弹确认框
- 若 `risk_stage_key` 未变化但 `system_version_hash` 变化：
  - 未编辑 proposal：可被新版本替换
  - 已编辑 proposal：不覆盖，只提示“模型有新建议”
- 若风险从 `Orange/Red` 回落到 `Yellow/Blue`：
  - 未确认 proposal 自动 `withdrawn`
  - 统一确认对话框关闭

### 7.5 已确认与未确认 proposal

- 未确认 proposal 可以被新版本替换或撤回
- 已确认 proposal 冻结，不再被后续版本覆盖
- 已确认 proposal 进入执行和留痕闭环

## 8. 全局确认对话框

### 8.1 弹出方式

- 当系统出现新的待确认区域 proposal 时，应立刻全局弹出
- 弹框直接打断当前界面

### 8.2 暂不处理

用户可以先关闭弹框，但：

- proposal 仍保持未确认
- 若没有明显变化，则 5 分钟后再次弹出
- 若 queue 版本或 proposal 内容变化，则可提前重弹

### 8.3 对话框展示内容

当前全局对话框优先展示：

- `action_display_category`
- `action_display_name`
- `action_display_tagline`
- 触发原因
- 建议内容
- 关联高风险对象
- 证据摘要 / grounding 摘要
- 可编辑的 `action_scope`

说明：

- `title` 退居次级信息
- 原始 `action_type` 不再作为主展示内容

## 9. 权限与编辑规则

### 9.1 权限边界

- 只有 `commander` 可以确认或驳回区域 proposal
- 只有 `commander` 可以编辑区域 proposal 草稿
- 其他角色默认只读

### 9.2 编辑范围

`commander` 当前可以编辑 `action_scope`，不能修改动作类型。

### 9.3 草稿保存规则

当前系统只保留 proposal 的最终草稿内容，不维护“系统原稿 + 人工草稿”双版本。

但会保留以下标记：

- `edited_by_commander`
- `last_editor`
- `has_new_system_suggestion`

### 9.4 编辑冲突

若 `commander` 已编辑草稿但尚未确认：

- 新的模型建议不能直接覆盖当前草稿
- 系统只设置“模型有新建议”提示

## 10. 执行闭环

### 10.1 当前目标

当前系统的执行闭环仍是“系统内执行”，不直接下发外部真实系统。

### 10.2 批准后的处理

批准区域 proposal 后：

- 由 `NotificationGateway` 调用 LLM 生成执行材料
- 生成 `NotificationDraft`
- 生成 `ExecutionLogEntry`
- 写入审计记录

### 10.3 输出元数据

当前执行材料和执行日志已支持：

- `generation_source`
- `model_name`
- `grounding_summary`

## 11. 当前字段要求

区域 proposal 当前至少应包括：

- `proposal_id`
- `proposal_scope`
- `action_type`
- `execution_mode`
- `action_display_name`
- `action_display_tagline`
- `action_display_category`
- `status`
- `title`
- `summary`
- `trigger_reason`
- `recommendation`
- `evidence_summary`
- `high_risk_object_ids`
- `action_scope`
- `risk_stage_key`
- `system_version_hash`
- `generation_source`
- `model_name`
- `prompt_profile`
- `grounding_summary`
- `edited_by_commander`
- `last_editor`
- `has_new_system_suggestion`
- `created_at`
- `updated_at`

## 12. 当前非目标范围

- 让模型直接主判区域风险等级
- `commander` 修改动作类型
- proposal 批准后直连外部真实执行系统
- 在系统中并行保留系统原稿与人工草稿双版本
- 让对象级追问默认进入区域级可确认 proposal 队列

## 13. 验收要点

当前实现至少应满足：

- 模拟结果更新并进入 `Orange/Red` 时，系统能主动生成区域 proposal
- 区域动作允许开放式 `action_type`
- 前端优先显示 `action_display_name/tagline/category`
- 新阶段或明显变化时，未确认 proposal 能正确替换并重弹
- 用户关闭弹框后，若仍未处理，5 分钟后自动重弹
- 高风险解除后，未确认 proposal 自动撤回
- 只有 `commander` 能确认、驳回和编辑草稿
- 对象级追问默认只返回建议，不产生新的区域级待确认动作
- proposal 批准后，系统能生成通知稿与执行日志，并写审计记录
