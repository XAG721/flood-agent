# 洪水预警系统前后端与总体功能总结

## 1. 文档目的

本文用于从当前代码实现出发，总结项目的系统定位、总体架构、前端组成、后端组成以及系统已具备的核心功能，便于用于汇报、交接和论文材料整理。

本文描述以当前 `现有平台` 原型实现为主，同时结合现有设计文档中的业务目标进行归纳。整体上，这是一套面向区县级洪水预警与应急指挥场景的原型系统，强调“确定性风险判断 + 多智能体辅助决策 + 人在回路审批 + 运行留痕”。

## 2. 系统总体定位

系统面向洪水或城市内涝场景，目标是在事件发生前后提供一套完整的辅助决策链路，覆盖：

- 洪水风险感知与事件启动
- 重点对象影响评估
- 主动预警与主动请示
- 多智能体协同分析与方案生成
- 指挥员审批与执行闭环
- 数据治理、审计与可靠性管理
- 事后日报、阶段总结、长期记忆沉淀

从项目现状看，当前代码重点落在以下能力上：

- 基于观测数据和模拟结果进行风险判定
- 在高风险阶段主动生成区域级待确认方案
- 通过 Copilot 对话方式支持指挥员查看、追问和确认
- 对对象画像、资源状态、RAG 文档和数据集进行运行时管理
- 保留审计、归档、复盘、评测和智能体回放能力

补偿相关内容在项目设计目标中占有重要位置，仓库中也保留了补偿输入数据和设计文档；但当前 `现有平台` 运行界面与接口的核心重心，主要集中在主动预警、主动请示和多智能体协同决策链路。

## 3. 总体架构

系统可以概括为“四层一辅端”结构：

### 3.1 表现层

- `frontend/`：主业务前端，提供值守控制台
- `3D_visual/`：三维场景展示端，用于 GLB 城市场景在 Cesium 中的定位、校准与查看

### 3.2 接口层

- `flood_system/api.py`：基于 FastAPI 对外暴露 REST API 与 SSE 流式接口
- 前端统一通过 `/api` 前缀访问后端能力

### 3.3 业务编排层

- `flood_system/system.py`：系统容器，负责组装仓储、RAG、数据服务、监督循环和报告服务
- `flood_system/platform/platform.py`：生产平台核心，串联事件、风险、暴露分析、决策、通知、提案、智能体和记忆能力
- `flood_system/platform/`：包含风险引擎、暴露分析、决策引擎、Copilot 编排、多智能体监督、通知网关、记忆存储等模块

### 3.4 数据与知识层

- SQLite：运行时主存储，保存事件、提案、日志、画像、资源状态、审计记录等
- `data_sources/`：原始与标准化数据源
- `flood_system/rag_raw/`：政策、案例、画像三类 RAG 原始文档
- `data/`：运行数据库、运行时 RAG 文件和测试数据

### 3.5 辅助能力

- 数据集构建与校验流水线
- 审计与归档
- 日报、阶段总结、长期记忆
- 评测与任务回放

## 4. 前端总结

## 4.1 主业务前端

主前端位于 `frontend/`，技术栈为：

- React 18
- TypeScript
- Vite
- `react-router-dom`
- `framer-motion`

前端本质上是一个“应急值守控制台”，以 `useV2OperatorConsole` 为核心状态管理入口，统一负责：

- 启动和恢复事件会话
- 拉取风险态势、对象影响和智能体状态
- 发送 Copilot 消息
- 订阅区域待审批提案队列
- 调用审批、编辑草稿、数据管理、评测和归档等操作

### 4.1.1 页面构成

主前端包含五个主要业务页面：

- 总览页 `Overview`
  - 展示当前事件概况、风险趋势、重点对象和信号时间线
- 指挥 Copilot 页 `Copilot`
  - 通过对话方式查看分析结果、追问风险、处理建议
- 行动处置页 `Operations`
  - 展示执行流程、区域分析包、提案历史、待确认队列和处置建议
- 数据管理页 `Data`
  - 管理对象画像、区域资源、事件资源、RAG 文档和数据集流水线
- 智能体页 `Agents`
  - 查看智能体状态、任务、共享记忆、触发事件、监督运行和评测结果
- 可靠性页 `Reliability`
  - 查看审计、告警、归档状态和权限治理

### 4.1.2 前端交互特点

当前前端的主要交互特点包括：

- 以“事件 + 会话”作为主线，而不是单纯页面切换
- 通过 SSE 实时订阅待审批区域 proposal 队列
- 使用全局弹窗打断式提醒新的待确认方案
- 支持按角色切换操作员权限，体现“人在回路”的审批边界
- 把对象影响评估、智能体状态、执行记录和治理信息放在同一套控制台中统一查看

## 4.2 三维展示前端

`3D_visual/` 是独立的三维展示端，技术栈为：

- React
- TypeScript
- Vite
- Cesium

其主要职责不是业务审批，而是三维场景的展示和校准，具体包括：

- 加载 CityEngine 导出的 GLB 模型
- 根据锚点经纬度和地形高程完成模型落位
- 调整缩放、姿态、偏移和视角
- 切换地形、影像、锚点和调试显示
- 作为空间可视化辅助手段服务于汇报或场景演示

因此，整个项目的前端并不是单一页面应用，而是“主业务控制台 + 三维场景展示端”的组合。

## 5. 后端总结

## 5.1 技术与组织方式

后端位于 `flood_system/`，采用 Python + FastAPI 实现，整体风格是“单体式原型系统 + 明确分层模块”。

核心后端组成如下：

- `api.py`
  - 暴露 HTTP 接口与 SSE 流接口
- `system.py`
  - 负责组装系统容器和后台服务
- `repository.py`
  - 基于 SQLite 的数据读写层
- `data_pipeline/`
  - 数据集抓取、构建、验证和同步
- `现有平台/`
  - 承载当前 现有平台 核心业务逻辑

## 5.2 现有平台 核心能力模块

`flood_system/platform/` 是当前系统最核心的业务层，主要包含：

- `hazard_engine.py`
  - 风险计算与风险等级生成
- `exposure_engine.py`
  - 重点对象影响评估
- `decision_engine.py`
  - 决策与策略生成约束
- `copilot_orchestrator.py`
  - 对话式 Copilot 编排
- `regional_proposals.py`
  - 区域提案生成、版本替换与状态管理
- `notification_gateway.py`
  - 审批后执行材料与通知草稿生成
- `multi_agent.py`
  - 智能体监督循环、后台清理等服务
- `memory_store.py`
  - 会话记忆与运行记忆存储
- `reporting.py`
  - 日报、阶段总结、长期记忆与复盘相关能力
- `security.py`
  - 角色权限与动作边界控制

## 5.3 后端接口能力

当前后端接口大体可以分为八类：

- 系统健康与权限
  - 健康检查、角色能力查询、监督循环状态
- 事件与风险
  - 创建事件、接收观测、接收模拟更新、获取风险态势
- 提案与审批
  - 查看待审批 proposal、编辑草稿、审批、驳回、区域分析包审批
- Copilot 与建议生成
  - 启动会话、发送消息、生成对象建议、查看记忆
- 运行时管理
  - 对象画像管理、资源状态管理、RAG 文档导入与重载
- 数据工程
  - 数据源抓取、构建、验证、同步、任务取消与重试
- 智能体治理
  - 智能体状态、任务时间线、共享记忆、触发事件、监督运行、任务回放
- 可靠性与评测
  - 告警、审计、归档、评测、报告重放

## 5.4 存储与后台服务

后端当前使用 SQLite 作为运行时核心存储，适合原型验证、实验复现和本地演示。

系统还会启动若干后台服务：

- 监督循环服务
  - 定时驱动智能体监督逻辑
- Housekeeping 服务
  - 归档和清理历史记录
- 日报服务
  - 生成事件日报
- 复盘服务
  - 沉淀长期记忆和阶段性总结

## 6. 系统总体功能总结

从用户视角看，系统当前具备以下总体功能：

### 6.1 洪水事件管理

- 创建洪水事件
- 接入观测数据与模拟结果
- 更新事件风险状态
- 记录事件全过程时间线

### 6.2 风险判定与影响分析

- 根据观测或模拟结果生成区域风险等级
- 对学校、社区、养老机构、居民点等重点对象进行影响评估
- 识别高风险对象、影响时间和风险原因

### 6.3 主动预警与主动请示

- 在进入高风险阶段后自动触发区域级动作建议
- 生成待确认的区域 proposal
- 对 proposal 进行排队、替换、撤回和状态跟踪
- 通过全局弹窗提醒指挥员进行确认

### 6.4 Copilot 对话式决策辅助

- 围绕事件进行多轮中文问答
- 解释风险原因、重点对象影响和建议动作
- 结合上下文持续生成处置建议
- 在对话中接入待审批 proposal 的查看与处理链路

### 6.5 指挥审批与执行闭环

- 指挥员可编辑 proposal 草稿中的动作范围
- 指挥员可批准或驳回 proposal
- 批准后自动生成执行材料、通知草稿和执行日志
- 全过程保留审计记录

### 6.6 数据治理与运行时维护

- 维护对象画像
- 维护区域和事件资源状态
- 导入和重载 RAG 文档
- 抓取、构建、校验并同步数据集

### 6.7 智能体协同与治理

- 展示智能体状态和任务结果
- 查看共享记忆、触发事件和时间线
- 手动运行监督循环
- 回放智能体任务与评测报告

### 6.8 审计、归档与复盘

- 查询开放告警和审计记录
- 执行归档
- 生成日报与阶段总结
- 沉淀长期记忆，为后续事件提供经验支持

### 6.9 三维场景辅助展示

- 展示 CityEngine 城市场景模型
- 对空间场景进行定位、校准和视角切换
- 为演示和汇报提供更直观的空间表达

## 7. 典型业务链路

下面是系统当前最核心的一条业务主链路：

1. 创建洪水事件。
2. 接收观测数据或模拟数据更新。
3. 后端计算风险状态并更新事件等级。
4. 高风险阶段触发智能体监督与区域提案生成。
5. 前端通过 SSE 收到新的待审批 proposal 队列。
6. 指挥员在控制台或全局弹窗中查看 proposal。
7. 指挥员编辑、批准或驳回 proposal。
8. 批准后生成通知草稿、执行记录和审计留痕。
9. 事件推进过程中持续沉淀日报、阶段总结和长期记忆。

这条链路体现了系统当前最重要的设计思想：风险判断尽量确定化，建议生成尽量智能化，最终动作必须由人确认。

## 8. 项目特点与当前阶段判断

结合代码和文档，当前系统具备以下特点：

- 业务链路完整，已经不是单点功能演示
- 前后端分工清晰，控制台能力较完整
- 智能体能力不是独立聊天，而是嵌入到应急流程中
- 具备数据治理、审计、评测和回放等研究型系统特征
- 适合作为论文原型、答辩演示系统和功能验证平台

同时也应明确其当前阶段属性：

- 更偏研究原型和实验平台，而非生产级政务系统
- 当前以本地 SQLite 和本地数据组织为主
- 三维端主要用于可视化辅助，不承担主业务流程
- 补偿相关能力在整体设计中占比很高，但当前 现有平台 运行主链路更聚焦预警、请示和协同决策

## 9. 一句话总结

这套系统可以概括为：一套以 FastAPI + React 为主干、以多智能体协同和 RAG 为增强、以“主动预警、主动请示、人工审批、执行留痕”为核心闭环的洪水应急指挥原型系统。
