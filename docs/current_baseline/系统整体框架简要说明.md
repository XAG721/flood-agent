# 系统整体框架简要说明

## 1. 系统定位

本项目是一个面向碑林区场景的洪水预警与指挥平台，当前代码已经进入 `现有平台` 阶段。系统的核心定位不再是“回答问题的原型”，而是：

- 用规则引擎对模拟结果做确定性的区域风险判定
- 用 LLM 生成用户可见的问答、建议、区域动作、请示草稿和执行材料
- 用多代理和统一对话框把主动预警、主动请示与执行留痕串成闭环

## 3. 当前主链路

### 3.1 区域级主动预警与主动请示

1. 模拟结果写入 `POST /platform/events/{event_id}/simulation-updates`
2. 平台根据阈值计算区域综合风险
3. 发布 `simulation_updated` trigger
4. `PlanningAgent` 调 LLM 生成区域动作集合
5. `RegionalProposalManager` 为每条动作生成区域 proposal
6. 前端通过 SSE 收到 pending queue
7. 全局对话框向 `commander` 逐条请示
8. 批准后调用 `NotificationGateway` 生成执行材料与执行摘要

### 3.2 对象级问答与建议

1. 用户发起对象级问答或 advisory 请求
2. 平台获取 impact、知识证据、上下文
3. 调用 `DecisionEngine + llm_gateway`
4. 产出 advisory 或 Copilot 回答
5. 当前默认不把对象级建议注入区域级可确认 proposal 队列

## 4. 区域 proposal 当前模型

`ActionProposal` 目前已支持以下核心字段：

- `proposal_scope`
- `action_type`
- `execution_mode`
- `action_display_name`
- `action_display_tagline`
- `action_display_category`
- `risk_stage_key`
- `system_version_hash`
- `generation_source`
- `model_name`
- `prompt_profile`
- `grounding_summary`

这意味着系统已经从“固定动作模板”切到“开放式区域动作”：

- `action_type` 可以是开放值
- 前端优先使用展示字段做产品化呈现
- `execution_mode` 决定进入通知、转移、调度或通用行动闭环

## 5. Copilot 当前形态

当前 Copilot 不是简单问答，而是受控的 agentic 组件：

- `RulePlanner`
  - 负责基础工具与安全边界
- `LLMPlannerAdapter`
  - 在白名单内补充或重排计划
- `ExecutionEngine`
  - 负责工具执行、retry、replan、freshness
- `CopilotOrchestrator`
  - 负责 session、message、structured answer、memory 聚合

与早期版本相比，当前变化是：

- planner 仍存在，但最终回答正文已由 LLM 生成
- structured answer、对象建议、区域 proposal、执行材料都带 `generation_source/model_name/grounding_summary`

## 6. 记忆体系

当前记忆体系分为三层：

- `Session Memory`
  - 面向 Copilot 会话，保存 focus、goal、pending proposal 等
- `Event Shared Memory`
  - 面向事件协作，保存 active agents、decision path、top risks、shared actions 等
- `Operational Experience Store`
  - 面向跨事件经验，沉淀 proposal、route、advisory 等历史 outcome

## 7. 数据管线

碑林区数据管线分为四层：

- `raw`
  - 原始公开源缓存，按 source_id 建目录，保存原始文件、元数据、artifact 映射与版本索引
- `normalized`
  - 标准化 JSON 产物
- `bootstrap`
  - 系统冷启动直接使用的 CSV / JSON
- `runtime rag`
  - 运行期知识文档

当前已经支持：

- source 级抓取与重抓
- `cache_metadata.json / artifacts.json / versions.json`
- raw completeness 判断
- dataset job 历史、取消、重试
- 演示库同步
- raw cache health 前端可视化

当前交互的关键变化是：

## 10. 当前系统边界

- 当前仍以单体架构为主，强调可运行、可审计、可扩展
- 高风险动作不自动越过人工确认
- 多代理已经具备协作与 replay 能力，但仍保持受控拓扑
- 区域风险判定保持规则确定性，不交由模型主判
- 后续如需进一步扩展，可继续对接 Redis、Kafka、PostgreSQL 等外部设施
