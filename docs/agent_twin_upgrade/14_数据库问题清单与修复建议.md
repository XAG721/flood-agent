# 数据库问题清单与修复建议

## 1. 文档目的

本文档用于记录当前 `AgentTwin Flood` 主数据库的检查结果，并形成一份可执行的数据治理清单。

本次检查面向当前默认运行主库：

- 数据库路径：`data/flood_warning_system_v2.db`
- 检查时间：`2026-04-27`
- 检查目标：判断数据库中的演示数据是否足以支撑当前“生产级 demo 测试品”的展示、联调和闭环验证

本文档不修改数据库，仅输出问题、影响与修复建议。

## 2. 总体结论

当前数据库的**基础结构和核心关系基本自洽**，系统不至于因为明显的数据断裂而无法运行；但从“生产级 demo 测试品”的标准看，数据库内容**还不够合理**，主要存在以下 4 类问题：

1. 闭环业务数据缺失，无法完整支撑 `proposal -> 审批 -> warning -> 执行留痕`
2. 演示事件重复累积，且全部维持 `active`
3. 数据时效偏旧，削弱了系统的“实时感”和可信度
4. 智能体运行日志在单一事件上异常堆积，容易造成展示偏移和库体积膨胀

因此可以判断：

- 作为开发库或联调用库：`基本可用`
- 作为甲方展示用主库：`需要治理后再使用`

## 5. 问题清单

### P1. 闭环业务表为空

#### 现象

以下关键表当前均为 `0` 行：

- `v2_action_proposals`
- `v3_audience_warnings`
- `v2_notification_drafts`
- `v2_execution_logs`

#### 判断

这说明数据库里几乎没有真实落库的：

- proposal 草稿
- 审批结果
- warning 生成结果
- 执行留痕

#### 对 demo 的影响

- 首页右侧闭环指挥台虽然可以展示流程，但数据库缺少真实闭环样本
- `/operations`、`/reliability` 中与闭环相关的真实数据支撑不足
- 演示时如果切换事件或刷新主屏，可能只能依赖运行时即时生成，而不是稳定的历史样本

#### 结论

对于“生产级 demo 测试品”而言，这一项是**最高优先级问题**。

#### 修复建议

至少补齐 1 条完整链路样本，建议包含：

1. `1` 条待审批 proposal
2. `1` 条已批准 proposal
3. `2-4` 条 audience warning
4. `1-2` 条 execution log
5. 至少 `1` 条与 proposal 关联的 audit / closure 记录

#### 验收标准

- `v2_action_proposals` 行数大于 `0`
- `v3_audience_warnings` 行数大于 `0`
- `v2_notification_drafts` 或 `v2_execution_logs` 至少有一类大于 `0`
- 所有闭环数据都可以通过 `proposal_id` 串联起来

### P1. 活动事件重复且未关闭

#### 现象

`v2_events` 当前共有 `5` 条事件，并且 `5` 条全部是 `active`。

标题分布如下：

- `碑林区洪涝预警演示事件`：`3` 条
- `碑林区洪水高风险监测事件`：`2` 条

#### 判断

这更像是多次打开前端或多次初始化 demo 时重复创建了事件，但旧事件没有被归档或关闭。

#### 对 demo 的影响

- “当前事件”语义不干净
- 如果前端默认取“最近活动事件”，容易出现事件上下文漂移
- 后续如果演示切换会商记录、shared memory、agent 结果，可能混入旧事件数据

#### 修复建议

保留 `1` 个主事件作为当前演示事件，其余旧事件：

- 标记为 `closed` / `archived`
- 或迁移到历史库
- 或在展示环境初始化时统一清理

#### 验收标准

- 主库中 `active` 事件数应控制为 `1`
- 同类型演示事件不应长期重复累积

### P1. 数据时间过旧

#### 现象

在 `2026-04-27` 检查时，数据库中关键数据的最新时间约为：

- 事件最新更新时间：`2026-04-15T04:01:30+00:00`
- 风险态势最新生成时间：`2026-04-15T04:01:30Z`
- 智能体任务最新时间：`2026-04-16T05:44:37+00:00`
- 智能体结果最新时间：`2026-04-16T05:44:37+00:00`
- supervisor 健康状态最新时间：`2026-04-16T05:45:12+00:00`

#### 判断

当前主库已经约 `11-12` 天没有被刷新。

#### 对 demo 的影响

- 演示“实时数字孪生”时可信度不足
- 如果页面明确展示时间标签，容易被直接看出是陈旧数据
- 甲方可能质疑系统是否真的在持续运行

#### 修复建议

在演示前刷新一轮数据库，至少更新：

- 当前主事件
- hazard state
- shared memory
- supervisor health state
- 最近一次会商和审计时间

#### 验收标准

- 演示主库的关键时间应在演示日前 `24-48` 小时内
- 首页、可靠性页、会商页中不出现明显陈旧时间戳

### P2. 单一事件智能体日志堆积过多

#### 现象

`v2_agent_results` 共 `6089` 条，其中：

- `event_4d50ac557f`：`5857` 条
- 其余 4 个事件合计：`232` 条

`v2_agent_tasks` 的分布也类似：

- `event_4d50ac557f`：`5857`
- `event_5932c0b94b`：`156`
- `event_6d203ffa03`：`49`
- `event_1de79164f1`：`16`
- `event_18b279f412`：`12`

#### 判断

某个事件明显经历了大量 supervisor / agent 循环。技术上不一定错误，但对展示库来说已经失衡。

#### 对 demo 的影响

- 查询最近结果时，容易被单一事件“淹没”
- 库膨胀较快
- 如果后续展示智能体结果对比页，会给人“系统一直在重复跑同一个事件”的观感

#### 修复建议

- 对旧事件日志做归档或裁剪
- 只保留最近 `N` 轮 supervisor / agent 结果
- 演示库中每个事件保留有限、可讲解的结果样本

#### 验收标准

- 单个演示事件的 agent 结果数量可控
- 最近事件与历史事件的数据量差异不至于数量级失衡

### P3. 区域资源状态过于简化

#### 现象

`v2_area_resource_status` 仅有 `1` 条区域级资源记录。

#### 判断

这对开发和基础联调够用，但对“生产级 demo”来说，资源层略显单薄。

#### 对 demo 的影响

- 行动处置台里的资源调度感不够强
- 难以体现“不同事件/不同区域/不同时间”的资源变化

#### 修复建议

可选增强项：

- 增加 `event_resource_status`
- 增加按时间片的资源变化样本
- 增加关键资源短缺/恢复的对比记录

#### 验收标准

- 至少能够展示“资源充足 / 资源紧张 / 资源恢复”三种状态之一

## 6. 当前数据中相对合理的部分

以下内容本次检查认为基本合理：

### 6.1 事件与风险态势可以一一对应

`v2_events` 与 `v2_hazard_states` 数量同为 `5`，抽样中未发现事件缺少风险态势的情况。

### 6.2 智能体任务、结果、任务事件没有明显孤儿记录

检查中未发现：

- `agent result` 指向不存在任务
- `task event` 指向不存在任务

说明智能体运行数据的基本引用关系是成立的。

### 6.3 对象画像人数没有明显越界

`v2_entity_profiles` 中：

- `profile_count = 8`
- `current_occupancy = 5464`
- `resident_count = 6412`

未发现 `current_occupancy > resident_count` 的明显异常。

### 6.4 对象画像内容具备展示价值

抽样对象包括：

- 老人家庭
- 小学
- 生物制剂厂
- 中心医院
- 医养中心

这类对象对演示“影响链、脆弱性、优先级差异”是有价值的。

## 7. 修复优先级建议

建议按以下顺序处理：

### 第一优先级：必须处理

1. 清理重复 `active` 事件，只保留 `1` 个当前主事件
2. 生成并落库 `proposal -> 审批 -> warning -> execution` 最小闭环样本
3. 刷新主事件和风险态势时间戳

### 第二优先级：强烈建议处理

1. 裁剪 `event_4d50ac557f` 过量智能体日志
2. 保留 `1-2` 个历史事件作为“复盘/历史样本”

### 第三优先级：增强项

1. 丰富资源状态层
2. 增加更多闭环后的通知和执行留痕样本
3. 为 reliability / operations 页面准备更完整的治理数据

## 8. 建议的目标库形态

建议为甲方展示准备一份“干净主库”，形态如下：

- `1` 个当前活动事件
- `1-2` 个已关闭历史事件
- `8-12` 个重点对象画像
- `1` 套完整的 hazard / shared memory / agent council 轨迹
- `1-2` 条 pending proposal
- `1` 条 approved proposal
- `2-4` 条 audience warnings
- `1-3` 条 execution logs
- `1` 组可讲解的 audit / closure 记录

这样既能支撑首页，也能支撑 `/operations`、`/agents`、`/reliability` 的演示。

## 9. 最终结论

当前数据库不是“坏库”，但它更像一份**长期开发与调试积累库**，而不是一份**经过治理、可直接用于甲方展示的演示主库**。

最核心的问题不是结构损坏，而是：

- 闭环数据缺失
- 活动事件重复
- 时效偏旧
- 智能体运行日志失衡

只要先完成一轮数据库治理，就可以显著提升当前系统的演示可信度和产品完成度。
