# AgentTwin Flood V3 系统架构设计

## 1. 架构目标

在不推翻当前 `V2` 单体原型的前提下，引入一层 `V3 AgentTwin Decision Layer`，把现有系统从“风险结果驱动”升级为“影响链与行动推演驱动”。

同时，前端展示层应从“V2 控制台”升级为：

**数字孪生智能体洪水预警系统前端**

这意味着前端层不仅是数据展示层，也承担数字孪生态势表达、对象联动、智能体对话控制和审批入口承接。

## 2. 当前基线

当前代码基线主要由以下模块构成：

- 接口层：`flood_system/api.py`
- 应用装配：`flood_system/system.py`
- 数据存储：`flood_system/repository.py`
- `V2` 业务层：`flood_system/v2/`
- 前端入口：`frontend/src/App.tsx`
- 前端状态编排：`frontend/src/hooks/useV2OperatorConsole.ts`

## 3. 目标架构

```text
前端展示层
-> 业务接口层
-> V3 AgentTwin 决策层
-> V2 平台能力复用层
-> 数据与知识层
-> 治理与审计层
```

## 4. 分层说明

## 4.1 前端展示层

职责：

- 展示数字孪生城市洪涝态势主屏
- 展示影响链、候选方案、会商结果和分众预警
- 承接对象选择、智能体对话控制、proposal 审批和证据查看

基于当前前端的承接点：

- `frontend/src/App.tsx`
- `frontend/src/components/GlobalRegionalProposalDialog.tsx`
- `frontend/src/pages/OverviewPage.tsx`
- `frontend/src/pages/OperationsPage.tsx`
- `frontend/src/pages/AgentsPage.tsx`

前端层建议新增或重构的核心能力：

- 数字孪生城市画布
- 右侧决策带
- 底部对象带
- 智能体对话控制弹窗
- 行动处置编排台
- 智能体协同透明页

## 4.2 业务接口层

职责：

- 保留 `/v2/*`
- 新增 `/api/v3/*`
- 提供 REST + SSE/WebSocket

建议新增：

- `POST /api/v3/weather-events/parse`
- `POST /api/v3/events/{event_id}/impact-chains/generate`
- `POST /api/v3/events/{event_id}/scenarios/generate`
- `POST /api/v3/events/{event_id}/agent-council/run`
- `POST /api/v3/proposals/{proposal_id}/warnings/generate`

前端接口设计要求：

1. 支持总览主屏拉取态势数据。
2. 支持对象选择后的详情联动。
3. 支持智能体对话控制请求与返回。
4. 支持 proposal 状态实时更新。

## 4.3 V3 AgentTwin 决策层

建议新增目录：

```text
flood_system/v3/
  weather_event_parser.py
  impact_ontology.py
  impact_graph.py
  impact_chain_engine.py
  action_template_store.py
  scenario_decision_lab.py
  action_scorer.py
  agent_roles.py
  agent_council.py
  agent_memory.py
  evidence_verifier.py
  audience_warning_center.py
  cap_like_alert.py
  message_consistency_checker.py
  approval_policy.py
  decision_memory.py
```

职责：

- 事件解析
- 影响链生成
- 多方案推演
- 多智能体会商
- 分众预警
- 审批边界控制

## 4.4 V2 平台能力复用层

复用模块：

- `regional_proposals.py`
- `notification_gateway.py`
- `memory_store.py`
- `reporting.py`
- `security.py`

复用原则：

- `V3` 负责分析与生成前链路
- `V2` 继续承接 proposal、通知、审计和执行日志落库

## 4.5 数据与知识层

现状：

- SQLite 作为主存储
- `rag_raw/` 作为文档知识源
- `data_sources/` 作为原始与标准化数据来源

升级后新增重点：

- 影响链数据
- 动作模板库
- 智能体会商记录
- 分众预警记录
- 复盘经验库

## 4.6 治理与审计层

复用当前治理基础：

- `flood_system/v2/security.py`
- `flood_system/v2/multi_agent.py` 中的 supervisor、trigger、timeline、audit
- `flood_system/repository.py` 中的 `v2_audit_records`、`v2_operational_alerts`

新增目标：

- 校验证据引用
- 校验消息一致性
- 校验审批越权
- 记录模型版本、Prompt 版本和工具调用

## 5. 调用关系

### 5.1 主链路

1. `weather_event_parser`
2. `impact_chain_engine`
3. `scenario_decision_lab`
4. `agent_council`
5. `evidence_verifier`
6. `regional_proposals / notification_gateway`
7. `audience_warning_center`

### 5.2 前端链路

1. 总览大屏加载态势数据
2. 用户选择重点对象
3. 前端打开智能体对话控制台
4. 智能体返回解释、建议和 proposal 入口
5. 用户进入审批与处置闭环

## 6. 部署建议

### 6.1 开发阶段

- 单体 FastAPI + SQLite
- React 前端本地开发
- 3D 或地图表现层可独立构建

### 6.2 后续可扩展

- SQLite -> PostgreSQL
- TriggerEvent -> Redis/Kafka
- SSE -> WebSocket
- RAG 文档 -> 向量库

## 7. 兼容策略

1. 不删除现有 `/v2/*` 接口。
2. `V3` 先以补充接口和补充表方式接入。
3. 前端允许重构页面骨架，不应只理解为“在 V2 控制台中新增几个面板”。
4. 总览页、智能体交互和审批入口应优先完成统一体验升级。
