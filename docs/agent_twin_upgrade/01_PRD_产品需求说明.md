# AgentTwin Flood V3 产品需求说明

## 1. 产品背景

当前系统已经实现 `V2` 级的“主动预警 + 主动请示 + 人工审批 + 执行留痕”闭环，但系统表达仍主要围绕风险等级、对象建议和区域 proposal 展开。

根据 [`AgentTwin-Flood-Requirements.md`](../../AgentTwin-Flood-Requirements.md)，下一阶段需要把系统升级为以“影响链、行动推演、智能体协同、分众预警、可审计 proposal”为核心的辅助决策系统。

其中，前端不应继续延续普通后台的表达方式，而应升级为：

**数字孪生智能体洪水预警系统前端**

产品目标不是替代人的决策，而是把“天气预警”稳定转化为“空间化影响对象、可执行行动建议、可审批 proposal 和可追溯通知输出”。

## 2. 产品目标

### 2.1 总目标

把系统从“洪水风险展示与建议生成平台”升级为“面向城市内涝场景的 AgentTwin 辅助决策平台”，并配套一个能够体现数字孪生和智能体交互优势的前端工作台。

### 2.2 本期目标

1. 从天气文本或人工输入中生成结构化天气事件。
2. 从事件生成影响链，识别受影响对象、影响后果和证据来源。
3. 基于影响链推演多种候选行动方案并排序。
4. 通过多智能体协同输出可审计 proposal。
5. 生成领导版、部门版、社区版、公众版等分众预警内容。
6. 形成“数字孪生主屏 + 智能体对话控制 + proposal 审批”一体化前端闭环。
7. 保留人工审批和全流程留痕。

### 2.3 非目标

1. 不把高精度水动力模拟作为本期核心。
2. 不直接接入生产级短信网关或政务审批系统。
3. 不允许模型绕过审批直接执行高风险动作。
4. 不在本期实现大规模实时视频识别或传感器复杂融合。

## 3. 用户角色

### 3.1 业务角色

- 指挥员：审批高风险 proposal，查看会商结果和分众预警草稿。
- 值班员：录入事件、查看态势、发起补充分析、跟进执行。
- 专业部门人员：查看部门任务单和处置依据。
- 社区或街道人员：接收巡查清单、居民提醒和执行反馈。
- 公众：接收一致、简洁、可信的风险提示。

### 3.2 系统角色

- Orchestrator：编排多智能体流程。
- WeatherAgent：解释天气事件。
- ImpactAgent：生成影响链。
- ActionAgent：推演行动方案。
- ResourceAgent：校验资源与约束。
- CommunicationAgent：生成分众消息。
- AuditAgent：校验证据、权限和一致性。

## 4. 核心场景

### 场景一：强降雨事件自动解析

用户输入一段气象预警文本，系统自动提取：

- 灾害类型
- 影响区域
- 时间窗口
- 预警等级
- 确定性

### 场景二：影响链推演

系统将天气事件与对象画像、历史案例、SOP 和 RAG 文档关联，输出：

- 受影响对象
- 主要影响
- 次生影响
- 证据链
- 置信度

### 场景三：多方案行动推演

系统围绕同一事件生成多套方案，例如：

- 标准防御方案
- 资源优先方案
- 脆弱群体优先方案

并给出排序依据。

### 场景四：多智能体会商与 proposal 生成

系统将结构化输入分发给各角色智能体，产出：

- 角色结论
- 会商摘要
- 审计结果
- 待审批 proposal

### 场景五：分众预警输出

同一 proposal 被改写为：

- 领导摘要
- 部门任务单
- 社区巡查提醒
- 公众提示

并完成一致性校验。

### 场景六：数字孪生主屏与对话控制

用户在数字孪生主屏中查看洪水态势、点击重点对象，并通过智能体对话控制台继续追问：

- 当前对象为什么高风险
- 有哪些证据支持
- 推荐方案之间如何取舍
- 哪个 proposal 应优先审批

## 5. 功能范围

## 5.1 后端功能

- `weather-event parser`
- `impact chain engine`
- `scenario decision lab`
- `agent council`
- `audience warning center`
- `evidence verifier`
- `approval policy`
- `decision memory`

## 5.2 前端功能

前端功能统一以“数字孪生智能体洪水预警系统前端”为目标定义：

- 数字孪生城市洪涝态势主屏
- 对象联动与影响链查看
- 智能体对话控制弹窗
- 会商与行动方案推演
- proposal 审批与执行编排
- 分众预警内容对比
- 证据链与审计查看

## 5.3 数据功能

- 对象画像维护
- 动作模板维护
- 资源约束维护
- RAG 文档管理
- 历史案例与复盘沉淀

## 6. 关键页面需求

### 6.1 总览页

总览页应被定义为“数字孪生城市洪涝态势主屏”，而不是普通总览卡片页。

需展示：

- 当前事件概况
- 数字孪生空间画布
- 风险热区与重点对象
- 路网与关键节点状态
- 智能体摘要
- 待审批事项
- 全局唤起智能体入口

### 6.2 影响链查看

系统应支持按“天气 -> 对象 -> 影响 -> 行动”查看链路，并可查看对应证据。

### 6.3 会商与推演页

展示角色智能体观点、冲突点、候选方案、评分和排序。

### 6.4 proposal 审批页

需展示：

- 建议动作
- 对应对象
- 责任部门
- 执行时限
- 审批等级
- 证据链

### 6.5 分众预警页

同屏查看多个受众版本，支持差异高亮和一致性确认。

## 7. 前端体验要求

1. 用户进入首页时，应首先感受到“数字孪生指挥中枢”，而不是普通后台。
2. 数字孪生空间画布应成为前端主视觉中心。
3. 智能体对话控制应成为全局主入口之一。
4. 主屏、对话框、审批和行动处置之间应形成连续交互闭环。
5. 页面风格应适合桌面工作台和答辩展示。

## 8. 验收标准

1. 能从天气文本生成结构化天气事件。
2. 能生成至少一条完整影响链。
3. 能生成至少三套候选行动方案并排序。
4. 能形成一份包含证据和审批边界的 proposal。
5. 能输出至少四类分众预警内容。
6. 能在前端主屏中完成对象联动、智能体追问和 proposal 入口跳转。
7. 高风险动作必须经过人工审批。
8. 全过程可追溯到输入、证据、模型、操作人和审批人。
