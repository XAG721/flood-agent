# 演示主库重建方案

## 1. 文档定位

本文档承接 [`14_数据库问题清单与修复建议.md`](./14_数据库问题清单与修复建议.md)，用于说明如何重建一份面向甲方展示、联调验证和生产级 demo 测试的干净主库。

本文档只描述方案，不直接修改数据库。

目标不是保留开发过程中的全部历史数据，而是构建一份稳定、可讲解、可回放、可验证的演示数据库，使当前 `AgentTwin Flood` 能稳定展示以下主链路：

`事件 -> 数字孪生态势 -> 重点对象 -> 智能体会商 -> proposal -> 人工审批 -> 分众预警 -> 执行留痕 -> 审计闭环`

## 2. 当前问题摘要

当前主库 `data/flood_warning_system_v2.db` 的主要问题是：

- 活动事件过多：`5` 个事件全部为 `active`
- 闭环数据缺失：`proposal / warning / notification / execution log` 相关表基本为空
- 数据时间偏旧：关键数据停留在 `2026-04-15` 到 `2026-04-16`
- 智能体日志失衡：单一事件积累了大量 agent task 与 result
- 展示库和开发调试库混在一起，缺少稳定演示基线

因此建议不要直接在原库上零散修补，而是重建一份可控的演示主库。

## 3. 重建原则

### 3.1 不覆盖开发库

不要直接覆盖 `data/flood_warning_system_v2.db`。

建议先生成一份新库：

- `data/flood_warning_system_demo.db`

确认演示链路稳定后，再决定是否把 API 默认库切换到该文件。

### 3.2 保留一条主链路

演示库不追求数据量大，而追求链路完整。

推荐只保留：

- `1` 个当前活动事件
- `1-2` 个历史事件
- `8-12` 个重点对象
- `1` 套完整风险态势
- `1` 套智能体会商轨迹
- `1-2` 条待审批 proposal
- `1` 条已批准 proposal
- `2-4` 条分众 warning
- `1-3` 条执行留痕
- `1` 组审计与复盘记录

### 3.3 前端展示优先

演示主库必须优先支撑首页和核心二级页：

- 首页 `/`：数字孪生主屏、对象联动、审批微流程、warning 生成入口
- `/operations`：行动处置、proposal 队列、执行结果
- `/agents`：多智能体会商、证据对照、编排结果
- `/reliability`：审计、降级、闭环与可信性说明

### 3.4 数据必须可解释

所有演示数据都应该能被讲清楚：

- 为什么这个对象高风险
- 为什么推荐这个动作
- 证据来自哪里
- 为什么需要人工审批
- warning 为什么分不同受众
- 执行闭环如何追溯

## 4. 推荐目标库结构

### 4.1 当前活动事件

建议仅保留一个当前活动事件：

- `event_id`：建议固定为 `event_demo_beilin_primary`
- `area_id`：`beilin_10km2`
- `title`：`碑林区强降雨内涝风险演示事件`
- `status`：`active`
- `current_risk_level`：`Red`
- `current_stage`：`Command`
- `trigger_reason`：`生产级 demo 演示场景初始化`

### 4.2 历史事件

建议保留 `1-2` 条历史事件作为复盘背景：

- 一个已完成闭环事件
- 一个曾经触发降级或审计阻断的事件

历史事件状态建议使用：

- `closed`
- `archived`

### 4.3 重点对象

建议保留当前已有对象画像，并适当补齐空间展示所需属性。

推荐对象类型：

- 学校：用于展示儿童群体、放学高峰、分流路线
- 医院：用于展示关键服务连续性
- 医养中心：用于展示高脆弱人群转移
- 地铁/交通枢纽：用于展示地下空间和交通阻断
- 工厂/冷链仓库：用于展示资产损失和停电风险
- 低洼社区网格：用于展示居民转移和基层通知

对象数量建议控制在 `8-12` 个。

### 4.4 风险态势

风险态势需要支撑首页三类展示：

- 总体风险等级
- 重点对象风险排序
- Cesium 地图上的空间状态

建议包含：

- `hazard_tiles`
- `road_reachability`
- `monitoring_points`
- `overall_risk_level`
- `trend`
- `freshness_seconds`

演示库中风险态势时间应更新到演示当天或演示前 `24-48` 小时内。

### 4.5 智能体会商数据

建议准备一组可读性强的会商结果。

至少包含以下角色：

- `ImpactAgent`
- `ActionAgent`
- `ResourceAgent`
- `WarningAgent`
- `AuditAgent`
- `Orchestrator`

每个角色需要有：

- 一句话摘要
- 置信度
- 引用证据
- 缺口信息
- 推荐下一步

### 4.6 Proposal 数据

建议准备至少两类 proposal：

- 待审批 proposal：用于主屏右侧审批微流程
- 已批准 proposal：用于直接生成 warning 和展示闭环完成态

proposal 必须关联：

- `event_id`
- `entity_id`
- `high_risk_object_ids`
- `evidence_summary`
- `execution_mode`
- `status`

### 4.7 Warning 数据

建议为已批准 proposal 生成 `2-4` 条 warning。

推荐受众：

- 领导版
- 部门版
- 社区版
- 公众版

每条 warning 应包含：

- `audience`
- `channel`
- `content`
- `grounding_summary`
- `proposal_id`
- `event_id`

### 4.8 执行留痕

建议准备 `1-3` 条 execution log。

示例类型：

- `warning_dispatched`
- `resource_assigned`
- `route_confirmed`
- `evacuation_started`

执行留痕必须关联 `proposal_id`，用于 `/reliability` 展示闭环追溯。

## 5. 重建流程

### Step 1：备份当前主库

先备份当前主库，避免误操作影响开发状态。

建议备份文件：

- `data/backups/flood_warning_system_v2_YYYYMMDD_HHMMSS.db`

### Step 2：创建演示库

创建新库：

- `data/flood_warning_system_demo.db`

创建方式建议复用现有初始化逻辑：

- 使用 `SQLiteRepository` 建表
- 调用当前系统启动时的数据初始化流程
- 再注入演示专用数据

### Step 3：注入基础对象与资源

注入：

- area profile
- entity profiles
- area resource status
- RAG documents
- notification templates

基础对象可以复用现有 `beilin_10km2` 数据，但要保证对象数量、名称和前端地图点位一致。

### Step 4：注入当前主事件

注入一个固定主事件：

- `event_demo_beilin_primary`

状态设为：

- `active`
- `Red`
- `Command`

### Step 5：注入风险态势

为主事件注入 hazard state。

要求：

- 总体风险为 `Red`
- 趋势为 `rapidly_rising`
- 包含可讲解的监测点
- 包含至少 `5` 条道路可达性状态
- 包含与重点对象对应的风险片区

### Step 6：注入智能体会商样本

注入一组 agent task、agent result、task event 和 shared memory。

建议只保留必要样本，不要模拟大量循环。

推荐数量：

- `agent_tasks`：`6-12`
- `agent_results`：`6-12`
- `agent_task_events`：`20-40`
- `event_shared_memory`：`1`

### Step 7：注入 proposal 样本

建议注入：

- `1` 条 `pending`
- `1` 条 `approved`

这样首页右侧既可以展示“待审批”，也可以展示“已批准后生成 warning”。

### Step 8：注入 warning 与执行留痕

基于已批准 proposal 注入：

- `2-4` 条 audience warning
- `1-3` 条 execution log

### Step 9：注入审计与可靠性样本

注入：

- 人工审批记录
- 审计记录
- 降级说明
- supervisor health state

用于 `/reliability` 展示系统可信性。

### Step 10：运行接口验收

演示库创建后，需要验证：

- 首页可以加载 twin overview
- 对象详情可以加载
- 对话控制台可以返回结构化响应
- proposal 队列存在
- approved proposal 可以生成 warning
- `/agents` 有会商数据
- `/reliability` 有审计与闭环数据

## 6. 推荐验收接口

建议至少检查以下接口：

- `GET /v3/events/{event_id}/twin-overview`
- `GET /v3/events/{event_id}/objects/{object_id}`
- `GET /v3/events/{event_id}/agent-council`
- `POST /v3/events/{event_id}/dialog`
- `POST /v3/events/{event_id}/proposals/generate`
- `POST /v3/proposals/{proposal_id}/warnings/generate`
- `GET /v3/events/{event_id}/stream`

同时确认 V2 桥接接口可用：

- proposal 审批
- notification draft 查询
- execution log 查询
- audit record 查询

## 7. 演示库验收标准

演示主库合格标准如下：

- 当前活动事件只有 `1` 个
- 首页可加载 `TwinOverviewView`
- 首页至少有 `5` 个地图对象
- 至少有 `1` 条 pending proposal
- 至少有 `1` 条 approved proposal
- 至少有 `2` 条 audience warning
- 至少有 `1` 条 execution log
- `/agents` 能展示多智能体会商差异
- `/reliability` 能展示审计与闭环状态
- 所有主要数据时间都在演示前 `24-48` 小时内
- 断开 LLM 或 SSE 时，前端仍能展示稳定降级态

## 8. 数据刷新策略

演示库可以采用两种刷新策略。

### 8.1 固定样本刷新

每次演示前重建同一份固定样本库。

优点：

- 稳定
- 可回放
- 适合客户演示

缺点：

- 不体现长期运行历史

### 8.2 增量样本刷新

保留历史事件，只刷新当前主事件。

优点：

- 更像持续运行系统
- 能展示复盘和历史案例

缺点：

- 需要更严格的数据归档规则

当前阶段建议使用固定样本刷新。

## 10. 风险与注意事项

### 10.1 不要把开发调试数据直接用于演示

开发库通常包含重复事件、过量日志和临时状态。直接用于演示会削弱系统可信度。

### 10.2 不要只刷新时间戳

仅更新时间戳但不补闭环数据，会让库看起来新，但仍然无法支撑完整业务演示。

### 10.3 不要把所有历史都删除

至少保留 `1-2` 个历史事件，方便展示复盘、审计和经验沉淀。

### 10.4 不要让演示主库依赖实时 LLM 成功

演示主库应自带稳定样本。真实 LLM 成功时可以增强展示，失败时仍应有结构化降级结果。

## 11. 建议实施顺序

建议按以下顺序推进：

1. 编写 `rebuild_demo_db.py`
2. 生成 `data/flood_warning_system_demo.db`
3. 注入当前活动事件和对象画像
4. 注入 hazard state 与地图对象状态
5. 注入 agent council 与 shared memory
6. 注入 proposal、warning、execution log
7. 运行接口验收
8. 启动前端检查首页、operations、agents、reliability
9. 冻结演示库作为可回放基线

## 12. 最终交付物

演示主库重建完成后，应形成以下交付物：

- `data/flood_warning_system_demo.db`
- `scripts/rebuild_demo_db.py`
- `scripts/inspect_demo_db.py`
- `scripts/start-demo.ps1`
- 数据验收报告
- 演示主事件 ID
- 演示对象清单
- 演示链路说明

这些交付物可以把当前系统从“开发调试状态”推进到“可稳定对外展示的生产级 demo 测试品”。
