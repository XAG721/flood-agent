# 17. 可交付重构说明

## 1. 重构目标

本轮重构面向“生产级 Demo 测试品”的持续交付要求，重点不是改产品功能，而是降低旧代码耦合、清晰化模块边界，并保证核心链路仍可运行、可测试、可继续扩展。

重构原则：

- 保持 `flood_system.api:app`、`/v2/*`、`/v3/*` 对外入口不变。
- 保持 `frontend` 为唯一正式前端入口，`3D_visual` 继续作为 Cesium 资源与校准参考。
- 不确定是否可删的内容不直接删除，只标记为待确认资产。
- 每轮拆分后必须运行后端编译、后端测试、前端构建和前端测试。

## 4. 当前 API 边界

V3 作为主屏与智能体链路的聚合读写层：

- `GET /v3/events/{event_id}/twin-overview`
- `GET /v3/events/{event_id}/objects/{object_id}`
- `GET /v3/events/{event_id}/agent-council`
- `POST /v3/events/{event_id}/dialog`
- `POST /v3/events/{event_id}/proposals/generate`
- `POST /v3/proposals/{proposal_id}/warnings/generate`
- `GET /v3/events/{event_id}/stream`

V2 继续作为审批、通知、执行日志、审计与历史兼容层：

- proposal 审批/驳回
- notification draft 与 execution log
- audit record 与 reliability 查询
- legacy copilot / supervisor 兼容接口

已清理或迁出：

- 旧 Overview 页面与旧解释卡片类组件不再作为主入口依赖。
- 大型 API 聚合文件已拆分。

- `.pytest_cache`、`__pycache__`、`frontend/dist` 等生成物由 `.gitignore` 控制，不应提交。

暂不直接删除：

- `docs/v2_baseline/`：仍可作为 V2/V3 差异说明资料。
- `flood_system/bootstrap_data/compensation_inputs/`：属于历史功能资产，需业务确认后再归档。

5. 继续优化 Cesium 包体，重点是资源懒加载、演示模式骨架屏和可观测加载状态。
