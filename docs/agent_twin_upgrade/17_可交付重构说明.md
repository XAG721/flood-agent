# 17. 可交付重构说明

## 1. 本轮重构目标

本轮重构不是产品功能重做，而是把当前 AgentTwin Flood 生产级 demo 从“功能堆叠可演示”推进到“结构边界清晰、可继续开发、可验证交付”的状态。

重构原则：

- 保持 `flood_system.api:app` 对外启动入口不变，避免破坏后端启动方式。
- 保持 `/v2/*` 和 `/v3/*` 对外 API 路径不变，避免破坏前端和测试。
- 先剥离高收益边界：配置、V3 router、SSE 基础设施、前端 V3 API 门面、前端派生状态。
- 不确定是否可删的旧模块先标记，不直接删除。
- 对已知生成物和本地临时目录，依赖 `.gitignore` 管控，避免进入仓库。

## 2. 已落地的新结构

### 2.1 后端

```text
flood_system/
  api.py                         # FastAPI app 装配入口，保留 V2 兼容路由
  config.py                      # 环境变量与运行配置
  http/
    v3_router.py                 # AgentTwin/V3 HTTP router
  infrastructure/
    sse.py                       # SSE 编码与流式事件基础设施
  schemas/
    v3.py                        # HTTP router 使用的 V3 schema import surface
  storage/
    schema.py                    # SQLite 表结构与索引定义
  v2/                            # 现有审批、通知、审计、执行闭环
    platform_audit.py            # V2 平台审计与告警操作
  v3/                            # AgentTwin 聚合服务与读模型
```

重构变化：

- `FLOOD_DB_PATH`、默认数据库路径和应用版本从 `api.py` 移入 `config.py`。
- `/v3/*` 路由从 `api.py` 移入 `http/v3_router.py`。
- V2 proposal SSE 与 V3 twin SSE 统一复用 `infrastructure/sse.py`。
- SQLite 建表 SQL 从 `repository.py` 移入 `storage/schema.py`，避免运行时存储类继续承载大段 schema 文本。
- V2 平台的审计记录与运行告警操作移入 `platform_audit.py` mixin，`ProductionPlatform` 保持原有方法名和调用语义。
- `api.py` 仍保留 V2 兼容路由，后续可按 admin / events / proposals / agents 继续拆。

### 2.2 前端

```text
frontend/src/
  api/
    agentTwinApi.ts              # V3/AgentTwin API 门面
  features/
    dataManagement/dataModels.ts # 数据维护页空档案、空资源状态工厂
  state/
    agentTwinSelectors.ts        # 主屏与会商页派生状态
  hooks/
    useAgentTwinConsole.ts       # AgentTwin 编排 hook，消费 agentTwinApi
  components/
    DigitalTwinImpactScreen.tsx  # 主屏视图，不再直接拼多源/图谱推理
    MultiAgentDesk.tsx           # 会商室视图，复用差异对照 selector
```

重构变化：

- `useAgentTwinConsole` 不再直接调用 `lib/api.ts` 的 V3 方法，而是通过 `api/agentTwinApi.ts`。
- `DigitalTwinImpactScreen` 的多源数据面板、影响链图谱派生逻辑移入 `state/agentTwinSelectors.ts`。
- `App.tsx` 与 `MultiAgentDesk.tsx` 的 agent 差异对照逻辑统一复用 `buildAgentDivergenceRows`。
- `useAgentTwinConsole` 的 V3 overview / council 刷新已去掉对整个 V2 base 对象的 effect 依赖，避免主屏在渲染期间重复拉取 `/v3/events/{event_id}/twin-overview` 与 `/agent-council`。
- 数据维护页的空对象档案、空资源状态工厂移入 `features/dataManagement/dataModels.ts`，为后续把 Admin/Data 页面整体迁出 `App.tsx` 做边界铺垫。这里避免使用 `features/data`，因为根 `.gitignore` 的 `data/` 规则会忽略同名目录。
- 已清理当前入口无静态引用的旧总览/解释组件：`OverviewPage.tsx`、`OverviewHero.tsx`、`CopilotContextPanel.tsx`、`PlannerExplainCard.tsx`，并移除 `App.tsx` 中不再使用的 `riskTone`。

## 4. API 与流式协议边界

### 4.1 V3 主链路

- `GET /v3/events/{event_id}/twin-overview`
- `GET /v3/events/{event_id}/objects/{object_id}`
- `GET /v3/events/{event_id}/agent-council`
- `POST /v3/events/{event_id}/dialog`
- `POST /v3/events/{event_id}/proposals/generate`
- `POST /v3/proposals/{proposal_id}/warnings/generate`
- `GET /v3/events/{event_id}/stream`

### 4.2 V2 桥接链路

- proposal 审批 / 驳回仍走 `/v2/proposals/{proposal_id}/approve|reject`
- proposal 队列 SSE 仍走 `/v2/proposals/stream`
- 通知、审计、执行留痕仍由 V2 平台服务落库

### 4.3 SSE 统一策略

后端统一使用 `infrastructure/sse.py` 输出：

```text
data: {json payload}

```

V3 流式事件继续保持小负载结构：`event_type + version + payload`，前端根据事件类型决定局部刷新或全量刷新。

## 5. 后续推荐重构顺序

1. 拆 `lib/api.ts`：把 V2 bridge、admin、dataset、reliability API 继续拆成独立门面。
2. 继续拆 `App.tsx`：下一步优先把 `AdminDesk`、`ProfileEditor`、`ResourceStatusEditor`、`DatasetPipelinePanel` 迁到 `features/dataManagement`。
3. 继续拆 `repository.py`：先按 V3 warning、audit、resource、dataset 分组，避免一次性大改。
4. 继续拆 `v2/platform.py`：按事件、proposal、agent、admin、reporting 服务分层。
5. 更新测试：把旧页面标题/按钮唯一匹配改成语义更稳定的 role/name 或区域内查询。
