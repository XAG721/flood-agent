# AgentTwin Flood V3 按模块实施顺序

## 1. 文档目标

本文件用于说明 `AgentTwin Flood V3` 的推荐实施顺序，解决“先做什么、后做什么、为什么这样排”的问题。

该顺序综合考虑：

- [`AgentTwin-Flood-Requirements.md`](../../AgentTwin-Flood-Requirements.md) 中的阶段建议
- `agent_twin_upgrade` 文档中的依赖关系
- 当前仓库已有 `V2` 基础
- 演示与答辩优先级

## 2. 实施总原则

1. 先做主链路，再做增强能力。
2. 先做数据和后端基础，再做前端表现层。
3. 前端先做可演示主屏，再做深度页面。
4. 任何时候都不绕开现有 `V2` 审批链路。

## 3. 推荐实施顺序

## 阶段 0：需求冻结与样例确定

目标：

- 固化演示场景
- 固化主链路验收标准
- 固化页面命名与产品口径

必须完成：

- 选择 1 个城市内涝主样例
- 确定重点对象清单
- 确定主屏展示内容
- 确定 proposal 审批范围

原因：

- 后续数据准备、Prompt 设计、页面展示都依赖这个样例

## 阶段 1：数据与知识底座

目标：

- 先让系统“有东西可推理”

先做模块：

1. 对象画像
2. 资源约束数据
3. SOP / 历史案例 / 部门职责 RAG 文档
4. `V3` 补充数据表

依赖关系：

- 影响链依赖对象画像和知识数据
- 方案推演依赖资源约束与动作模板
- 智能体会商依赖结构化输入

本阶段产出：

- 最小数据集
- 数据表结构
- RAG 文档集

## 阶段 2：后端基础引擎

目标：

- 先打通“事件 -> 影响链 -> 方案”

先做模块：

1. `weather_event_parser.py`
2. `impact_ontology.py`
3. `impact_graph.py`
4. `impact_chain_engine.py`
5. `action_template_store.py`
6. `action_scorer.py`
7. `scenario_decision_lab.py`

原因：

- 没有这一步，后面的智能体协同和前端主屏都没有核心内容可展示

本阶段完成标志：

- 给定天气文本，能生成事件、影响链和排序后的候选方案

## 阶段 3：智能体与审计能力

目标：

- 把“推理结果”升级为“可解释、可审计、可审批”的结构化输出

先做模块：

1. `agent_roles.py`
2. `agent_council.py`
3. `evidence_verifier.py`
4. `approval_policy.py`
5. `message_consistency_checker.py`

随后补：

6. `agent_memory.py`
7. `decision_memory.py`

原因：

- 智能体协同和审计控制是 `V3` 的关键升级点
- proposal 生成前必须先有会商和审计结果

本阶段完成标志：

- 能形成可审计 proposal 草案
- 能阻断无证据或越权建议

## 阶段 4：API 与桥接层

目标：

- 把 `V3` 能力以接口方式暴露给前端和现有链路

先做接口：

1. `/api/v3/weather-events/parse`
2. `/api/v3/events/{event_id}/impact-chains/generate`
3. `/api/v3/events/{event_id}/scenarios/generate`
4. `/api/v3/events/{event_id}/proposals/generate`

随后做：

5. `/api/v3/events/{event_id}/agent-council/run`
6. `/api/v3/proposals/{proposal_id}/warnings/generate`
7. `/api/v3/events/{event_id}/postmortem/generate`

并行补充：

- `twin_overview_updated`
- `focus_object_updated`

原因：

- 前端重构必须依赖稳定接口，而不是直接耦合后端内部模块

本阶段完成标志：

- `V3` 主链路能通过 API 调用
- `V2` proposal 审批仍可继续复用

## 阶段 5：前端 P0 主屏与对话控制

目标：

- 优先做出最能体现系统升级价值的界面

先做页面：

1. 总览大屏
2. 智能体对话控制弹窗

先做组件：

1. 顶部指挥条
2. 中央数字孪生画布
3. 右侧决策带
4. 对象联动状态
5. 全局唤起智能体入口

原因：

- 这是答辩和展示价值最高的部分
- 用户也最先感知到这里的升级

本阶段完成标志：

- 首页已不再是普通后台
- 用户可在主屏中选中对象并唤起智能体追问

## 阶段 6：前端 P1 处置与协同页面

目标：

- 补齐“从主屏进入执行”的闭环

先做页面：

1. 行动处置页
2. 智能体协同页

重点能力：

- proposal 队列查看
- 方案比较
- 工具调用摘要
- 智能体任务时间线
- open questions / blocked reasons

原因：

- 当主屏和对话弹窗已经可用后，下一步就是让用户能继续处置和理解系统内部协同

本阶段完成标志：

- proposal 可从主屏进入审批与执行编排
- 用户可查看智能体协同透明化信息

## 阶段 7：分众预警、审计与复盘

目标：

- 完成从审批到通知到复盘的完整闭环

先做模块：

1. 分众预警对比区
2. 证据链抽屉
3. 审计阻断说明
4. 复盘入口与摘要

原因：

- 这是“可执行、可审计、可学习”的最终闭环
- 对答辩和论文创新点支撑很强

本阶段完成标志：

- proposal 审批后可直接生成多版本预警
- 审计和复盘页面有完整展示

## 阶段 8：测试、稳定化与答辩打磨

目标：

- 把功能做成“能稳定演示”的系统

重点工作：

1. 后端单元测试
2. API 集成测试
3. 前端主屏联动测试
4. 智能体对话控制测试
5. 冒烟测试
6. 演示脚本固定化

原因：

- 没有稳定化，前面所有功能都难以在答辩或汇报中可靠呈现

## 4. 并行实施建议

以下模块适合并行推进：

### 可与阶段 2 并行

- 数据底座整理
- Prompt 草拟
- 前端页面线框设计

### 可与阶段 4 并行

- 前端总览大屏静态骨架
- 智能体对话弹窗静态样式

### 可与阶段 6 并行

- 分众预警界面设计
- 审计与复盘展示设计

## 5. 最小可演示顺序

如果时间有限，建议严格按以下最小顺序推进：

1. 数据样例
2. 事件解析
3. 影响链生成
4. 方案推演
5. 会商与审计
6. proposal 生成
7. 首页重构为数字孪生主屏
8. 智能体对话控制弹窗
9. proposal 审批接入
10. 分众预警生成

## 6. 不建议的顺序

以下顺序不建议采用：

1. 先做大量前端视觉细节，再补后端主链路
2. 先做复盘和长期记忆，再补影响链与 proposal
3. 先做复杂三维效果，再确认主交互链路
4. 在 `V3` 未稳定前就强依赖新接口替换全部 `V2` 流程

原因：

- 会导致展示好看但主链路不通
- 会增加返工成本
- 会拖慢 MVP 成型

## 7. 最终建议

推荐你把实施顺序简化理解为三句话：

1. 先让系统能推理。
2. 再让系统能审批。
3. 最后让系统看起来像一个真正的数字孪生智能体洪水预警系统前端。
