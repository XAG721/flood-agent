# AgentTwin Flood V3 业务流程设计

## 1. 设计原则

1. 风险判断尽量确定化，智能体输出尽量结构化。
2. 高风险动作必须经过审批，不允许模型越权执行。
3. 业务流程要兼容现有 `V2` 的事件、proposal、通知和审计主链路。

## 2. 总体业务闭环

```text
天气预警/人工输入
-> 事件解析
-> 影响链生成
-> 行动方案推演
-> 多智能体会商
-> 审计校验
-> proposal 生成
-> 人工审批
-> 分众预警
-> 执行留痕
-> 事后复盘
```

## 3. 关键流程

## 3.1 事件创建与解析流程

1. 值班员录入天气文本，或系统接收外部预警文本。
2. `WeatherEventParser` 解析灾害类型、区域、时窗和等级。
3. 系统生成结构化 `weather_event`。
4. 若解析失败，进入人工补录。

参与方：

- 值班员
- WeatherAgent
- 事件服务

## 3.2 影响链生成流程

1. 系统读取事件、区域画像、对象画像、历史案例和 SOP。
2. `ImpactChainEngine` 识别受影响对象。
3. 生成主影响、次生影响、证据链和置信度。
4. 输出结构化 `impact_chains`。

参与方：

- ImpactAgent
- RAG 服务
- 对象画像库
- 案例库

## 3.3 候选行动推演流程

1. 基于影响链匹配动作模板。
2. 生成多个候选行动方案。
3. `ActionScorer` 从时效、资源、脆弱群体保护、审批复杂度等维度评分。
4. 输出排序结果与不推荐原因。

参与方：

- ActionAgent
- ResourceAgent
- 动作模板库

## 3.4 多智能体会商流程

1. Orchestrator 向各角色智能体分发同一事件上下文。
2. 各智能体输出自己的结构化结论。
3. AuditAgent 检查地名、资源、权限、证据和冲突。
4. Orchestrator 汇总成最终建议和待审批 proposal。

参与方：

- Orchestrator
- WeatherAgent
- ImpactAgent
- ActionAgent
- ResourceAgent
- CommunicationAgent
- AuditAgent

## 3.5 proposal 审批流程

1. 系统生成 proposal 并进入待审批队列。
2. 前端弹出审批面板。
3. 指挥员查看建议、证据、对象和责任部门。
4. 指挥员批准、驳回或修改参数范围。
5. 审批结果写入审计与执行日志。

参与方：

- 指挥员
- 审批策略服务
- 审计服务

## 3.6 分众预警发布流程

1. `AudienceWarningCenter` 基于已审批 proposal 生成统一 Alert Object。
2. 按受众改写为领导版、部门版、社区版、公众版。
3. `ConsistencyChecker` 校验口径一致性。
4. 审批通过后进入通知中心。

## 3.7 复盘学习流程

1. 事件结束后生成高风险复盘。
2. 抽取有效动作、失败动作、协调缺口和复用规则。
3. 更新知识图谱、动作模板库、案例库和 prompt 经验。

## 4. 泳道视角

| 泳道 | 主要职责 |
|---|---|
| 系统 | 触发流程、保存数据、推送待办 |
| 值班员 | 创建事件、补录信息、查看初步结果 |
| 指挥员 | 审批 proposal、决定高风险动作 |
| 部门人员 | 接收任务单、回传执行情况 |
| 社区人员 | 执行巡查、居民提醒、反馈现场情况 |
| 智能体 | 生成结构化分析与建议 |

## 5. 与现有 V2 的衔接

当前 `V2` 已具备：

- 事件入口
- 风险更新
- proposal 待审批队列
- SSE 推送
- 审计、通知、执行日志

`V3` 的接入方式建议为：

1. 保留 `V2` 事件与 proposal 主链路。
2. 在 proposal 生成之前插入“影响链 + 会商 + 推演”步骤。
3. 在 proposal 批准之后插入“分众预警”步骤。
