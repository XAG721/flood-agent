# AgentTwin Flood V3 开发任务清单

## 1. 文档目标

本清单用于将 [`AgentTwin-Flood-Requirements.md`](../../AgentTwin-Flood-Requirements.md) 和 `agent_twin_upgrade` 文档包进一步落成可执行开发任务。

任务拆解遵循以下统一目标：

- 后端升级为 `V3 AgentTwin` 决策链路
- 前端升级为“数字孪生智能体洪水预警系统前端”
- 保留 `V2` 主链路并逐步桥接 `V3`

## 2. 任务拆解原则

1. 先打通最小闭环，再增强表现和精度。
2. 优先完成“事件解析 -> 影响链 -> 方案推演 -> proposal -> 审批 -> 分众预警”主链路。
3. 前端优先完成“总览大屏 + 智能体对话控制 + proposal 入口”。
4. 所有高风险动作必须保留人工审批。

## 3. 开发任务总览

### A. 需求冻结与演示样例

- `A1` 固化演示场景
  - 目标：确定 1 个可完整串联的城市内涝样例
  - 输入：强降雨文本、影响区域、重点对象、资源限制
  - 产出：演示样例描述、验收剧本

- `A2` 固化验收口径
  - 目标：统一“能演示什么算完成”
  - 产出：主链路验收步骤、页面验收点、接口验收点

### B. 数据与知识底座

- `B1` 设计并落地 `V3` 补充数据表
  - 重点：`impact_chains`、`action_templates`、`agent_deliberations`、`audience_warnings`

- `B2` 准备最小对象画像数据集
  - 重点：学校、小区、医院、下穿通道、地下空间、安置点

- `B3` 准备最小 RAG 文档集
  - 重点：SOP、历史案例、部门职责、预警模板

- `B4` 准备资源约束数据
  - 重点：车辆、人员、物资、道路可达性、避险点

### C. 后端 V3 基础骨架

- `C1` 新建 `flood_system/v3/` 目录结构
- `C2` 完成 `weather_event_parser.py`

### D. 智能体与审计链路

## 6. 里程碑建议

### M1：最小闭环跑通

完成标准：

- 能解析天气事件
- 能生成影响链
- 能生成候选方案
- 能形成 proposal
- 首页已具备数字孪生主屏雏形
- 可打开智能体对话控制弹窗

### M2：前后端完整串联

完成标准：

- proposal 审批链路跑通
- 分众预警生成跑通
- 行动处置页与智能体协同页可用
- 前端主屏与对话控制联动可用

### M3：答辩版完成

完成标准：

- 视觉风格统一
- 审计与复盘表现完整
- 演示脚本稳定
- 关键测试通过

## 7. 完成定义

每个开发任务完成时，至少应满足：

1. 文档中已有对应设计依据。
2. 代码或接口已落地。
3. 能被前端或后端主链路实际调用。
4. 至少具备一个对应测试或验收样例。
