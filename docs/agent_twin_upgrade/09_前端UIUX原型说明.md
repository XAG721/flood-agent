# AgentTwin Flood V3 前端 UI/UX 原型说明

## 1. 文档定位

本文件用于定义 `AgentTwin Flood V3` 的前端原型方向，产品定位明确为：

**数字孪生智能体洪水预警系统前端**

这里的三个关键词必须同时成立：

1. 数字孪生：前端主界面必须具备空间态势表达能力，核心视觉不应是普通后台卡片，而应是“城市洪涝态势主屏 + 数字孪生画布”。
2. 智能体：前端主交互不应停留在按钮点击和页面切换，而应引入“智能体对话控制”作为核心入口。
3. 洪水预警系统：前端所有页面都应围绕洪水事件研判、风险传播、行动方案、人工审批、分众预警和审计留痕展开。

因此，`V3` 前端不是对 `V2` 控制台做轻量补丁，而是应在保留现有数据链路的前提下，对产品表达方式、页面骨架和交互模式进行明显升级。

## 2. 设计目标

## 2.1 总体目标

将当前前端从“风险信息查看界面”升级为“可用于指挥、研判、答辩展示和演示的数字孪生智能体洪水预警系统前端”。

用户进入前端后，应能够在统一界面中完成以下任务：

1. 看见当前洪水事件、区域态势和重点对象分布。
2. 理解洪水影响如何沿空间对象和行动链路传播。
3. 通过智能体对话框向系统发出追问、推演、确认和解释请求。
4. 对候选行动进行查看、比较、审批和留痕。
5. 将审批结果转化为不同受众版本的预警输出。

## 2.2 体验关键词

- 数字孪生
- 态势感知
- 指挥中枢
- 智能体协同
- 空间联动
- 研判闭环
- 可追溯

## 2.3 与 V2 前端的区别

`V2` 前端更像多面板业务控制台，而 `V3` 前端应体现以下变化：

1. 主视觉从信息卡片切换为“数字孪生城市态势主屏”。
2. 主交互从页面跳转切换为“对象联动 + 智能体对话控制 + 审批闭环”。
3. 主信息从静态建议列表切换为“空间对象、影响链、行动方案、审批状态、智能体判断”的联动展示。
4. 产品风格从普通后台切换为适合演示与指挥的中控台界面。

## 3. 当前前端承接点

当前代码中可以继续承接升级的入口如下：

- `frontend/src/App.tsx`
- `frontend/src/hooks/useV2OperatorConsole.ts`
- `frontend/src/pages/OverviewPage.tsx`
- `frontend/src/pages/OperationsPage.tsx`
- `frontend/src/pages/AgentsPage.tsx`
- `frontend/src/components/GlobalRegionalProposalDialog.tsx`

说明：

1. 可继续复用现有数据获取、状态管理、审批逻辑和消息流。
2. 但页面结构、视觉语言和主要交互方式允许重做。
3. 不应被“保留现有卡片布局”这一前提约束。

## 4. 前端产品定义

## 4.1 产品名称建议

建议在文档标题、页面品牌标题、答辩 PPT 和演示界面中统一采用以下名称：

**数字孪生智能体洪水预警系统前端**

可选英文副标题：

`Digital Twin Agent-Based Flood Early Warning Frontend`

## 4.2 页面角色定义

整个前端不是若干独立页面的松散组合，而是一个有明确分工的“指挥工作台”。

建议页面角色如下：

- 总览大屏：城市洪涝态势主屏
- 行动处置：方案编排与审批闭环
- 智能体协同：智能体运行状态与推理透明化
- 数据底座：对象画像、资源状态和知识数据维护
- 审计复盘：证据链、审批留痕和复盘分析

其中，总览大屏必须成为最核心的页面。

## 5. 总体信息架构建议

## 5.1 一级导航

- 总览大屏
- 行动处置
- 智能体协同
- 数据底座
- 审计复盘

## 5.2 与现有页面映射

- `总览大屏` 对应当前 `Overview`
- `行动处置` 对应当前 `Operations`
- `智能体协同` 对应当前 `Agents`
- `数据底座` 对应当前 `Data`
- `审计复盘` 对应当前 `Reliability`

## 5.3 全局固定入口

建议所有页面统一保留以下全局入口：

1. 当前事件与风险等级条。
2. 当前值班角色切换控件。
3. 智能体状态信号区。
4. 待审批 proposal 数量提示。
5. “唤起智能体”全局按钮。

其中，“唤起智能体”不应只存在于单独页面，而应作为全局固定入口。

## 6. 核心原型设计

## 6.1 总览大屏：数字孪生城市态势主屏

这是 `V3` 前端最重要的页面，应明确采用“中央画布 + 左右情报带 + 底部对象带”的结构。

### 6.1.1 页面结构建议

推荐采用三层式布局：

1. 顶部指挥条
2. 中央数字孪生主画布
3. 左右侧情报与决策侧带

推荐结构如下：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部指挥条：事件 / 风险等级 / 智能体状态 / 全局操作          │
├───────────────┬────────────────────────────┬───────────────┤
│ 左侧态势带     │ 中央数字孪生城市画布        │ 右侧决策带      │
│ - 事件摘要     │ - 区域热力 / 涝点 / 河道     │ - 待审批动作     │
│ - 风险趋势     │ - 重点对象定位              │ - 智能体摘要     │
│ - 监测信号     │ - 路网与可达性              │ - 当前焦点对象   │
│ - 对象筛选     │ - 资源节点与避险点          │ - 建议行动与入口 │
├───────────────┴────────────────────────────┴───────────────┤
│ 底部对象带：学校 / 小区 / 医院 / 道路 / 安置点 / 桥梁等      │
└──────────────────────────────────────────────────────────────┘
```

### 6.1.2 中央数字孪生画布要求

中央区域不是装饰性背景，而是主信息容器，至少需要表达以下内容：

1. 洪水影响范围。
2. 重点对象的空间分布与风险强弱。
3. 路网、桥梁、泵站、医院、安置点等关键节点。
4. 通行状态、积水扩散、资源布置等动态信息。
5. 点击对象后的联动高亮。

若当前阶段暂时无法做到完整三维城市场景，可采用以下过渡形式：

- 2.5D 城市底图
- 风险热力图层
- 关键对象图标层
- 可达性路径层
- 洪涝扩散或波纹动画层

### 6.1.3 左侧态势带

左侧用于“先让用户理解局势”，建议包含：

- 当前洪水事件摘要
- 风险等级与趋势判断
- 时间窗口与影响区域
- 监测点异常信号
- 重点对象筛选入口
- 影响链入口

左侧不应只显示统计数字，而应承担“态势解释”的职责。

### 6.1.4 右侧决策带

右侧用于“让用户直接进入行动”，建议包含：

- 待审批 proposal 列表
- 智能体当前研判摘要
- 当前焦点对象卡片
- 推荐行动摘要
- 打开智能体对话框入口
- 进入行动处置页入口

右侧应让用户无需离开主屏，也能进入决策流程。

### 6.1.5 底部对象带

底部横向对象带建议用于快速切换重点对象，例如：

- 学校
- 小区
- 医院
- 道路节点
- 桥梁
- 安置点
- 地下空间

点击对象后，中央画布高亮位置，右侧更新对象详情，智能体对话上下文同步切换。

## 6.2 智能体对话控制弹窗

这是本系统区别于普通数字孪生大屏的关键交互，应作为前端主入口之一。

### 6.2.1 定位

它不是普通聊天框，而是：

**智能体指挥与追问控制台**

用户通过该弹窗可以：

1. 请求系统解释当前风险判断。
2. 围绕某个重点对象继续追问。
3. 要求系统生成或比较行动方案。
4. 要求系统解释 proposal 的依据、约束和风险。
5. 要求系统生成不同受众版本的预警文本。

### 6.2.2 触发方式

建议至少保留三种触发入口：

1. 顶部全局“唤起智能体”按钮。
2. 总览大屏右侧决策带按钮。
3. 点击重点对象后的快捷操作按钮。

### 6.2.3 弹窗结构建议

```text
┌────────────────────────────────────────────────────────┐
│ 标题区：当前对象 / 当前事件 / 智能体状态 / 关闭按钮    │
├────────────────────────┬───────────────────────────────┤
│ 左侧对话主区            │ 右侧辅助区                    │
│ - 对话历史              │ - 快速追问建议                │
│ - 智能体回答            │ - 当前焦点对象摘要            │
│ - 工具执行摘要          │ - 待审批 proposal             │
│ - 输入框                │ - 最新证据与行动建议          │
├────────────────────────┴───────────────────────────────┤
│ 底部：发送 / 继续追问 / 生成方案 / 查看依据            │
└────────────────────────────────────────────────────────┘
```

### 6.2.4 设计要求

1. 弹窗应支持大尺寸展示，可作为半屏工作台使用。
2. 用户不应离开总览页也能完成对话控制。
3. 弹窗中的对象上下文、事件上下文和待审批动作必须与主屏联动。
4. 对话内容应强调“解释、推演、建议、确认”，而不是泛化闲聊。

## 6.3 行动处置页

行动处置页是“从研判到执行”的编排台，而不是普通列表页。

### 6.3.1 页面职责

- 查看 proposal 队列
- 查看区域分析包
- 查看候选行动方案
- 对 proposal 执行批准、驳回、暂缓
- 查看执行链路和工具调用摘要

### 6.3.2 页面布局建议

建议采用“主流程板 + 右侧审批带”的结构：

- 左侧：流程编排、执行步骤、方案历史
- 右侧：待确认 proposal、时间线、对象关联、建议摘要

### 6.3.3 设计重点

该页面应强调“流程状态”和“审批边界”，而不是强调卡片数量。

## 6.4 智能体协同页

该页面用于回答两个核心问题：

1. 现在有哪些智能体在工作？
2. 它们为什么得出当前结论？

### 6.4.1 页面建议内容

- 智能体任务时间线
- 当前运行中的任务
- 任务状态与结果摘要
- 智能体之间的协同关系
- Open questions / blocked reasons
- Supervisor 状态

### 6.4.2 设计要求

1. 不要做成纯日志页。
2. 要让用户快速区分谁在分析、谁在等待、谁被阻塞、谁提出了方案。
3. 要体现 Agent 协同结构，而不是单列消息堆叠。

## 6.5 数据底座页

数据底座页用于支撑前台可信度和可维护性，建议包括：

- 对象画像管理
- 区域资源状态维护
- RAG 文档导入与刷新
- 数据源构建与同步

在视觉上可以比总览页收敛，但不建议退回传统灰白后台样式。

## 6.6 审计复盘页

审计复盘页用于回答：

- 为什么系统这样判断？
- 它引用了哪些证据？
- 谁批准了它？
- 后续能否追溯？

建议包含：

- 审计记录查询
- 证据链查看
- 归档状态
- 复盘摘要
- 模型版本 / Prompt 版本 / 工具调用留痕

该页面应强调可信、清晰、可追溯。

## 7. 关键交互链路

## 7.1 从总览大屏进入智能体对话控制

1. 用户进入总览大屏。
2. 在数字孪生画布中点击某个重点对象。
3. 右侧决策带更新对象摘要与建议。
4. 用户点击“唤起智能体”。
5. 系统弹出智能体对话控制台，并自动带入当前对象上下文。
6. 用户继续追问风险原因、证据、方案与审批建议。

## 7.2 从总览大屏进入审批闭环

1. 用户在主屏右侧看到新的待审批 proposal。
2. 点击 proposal 后查看对象、区域、责任部门、时限和依据。
3. 用户进入行动处置页，或在对话控制弹窗中继续追问。
4. 用户完成批准或驳回。
5. 系统同步刷新主屏状态与通知链路。

## 7.3 从主屏生成分众预警

1. 用户在总览页确认处置动作。
2. 打开智能体对话控制台，要求系统生成不同受众版本。
3. 系统生成领导版、部门版、社区版、公众版预警内容。
4. 用户比较差异并确认输出。

## 8. 视觉风格建议

## 8.1 视觉方向

整体风格建议不是“蓝色政务后台”，而是：

**数字孪生指挥中枢**

推荐视觉特征：

- 深色主背景
- 城市网格 / 雷达 / 热力 / 连线光效
- 高对比对象高亮
- 清晰的态势信息带
- 科技感与决策严肃性兼顾

## 8.2 色彩建议

- 主背景：深蓝黑 / 石墨蓝
- 风险色：蓝 / 黄 / 橙 / 红
- 智能体强调色：青蓝或琥珀
- 审批动作色：绿色通过、红色驳回、橙色待确认

不建议使用过于轻松或消费级产品风格的配色。

## 8.3 图形元素建议

- 城市网格
- 风险热区
- 路网连线
- 涝点脉冲标记
- 重点对象定位标识
- 波纹扩散动画
- HUD 辅助框

## 8.4 动效建议

动效应服务于态势表达，不应只是装饰。

推荐动效：

1. 风险区域呼吸式高亮。
2. 重点对象脉冲定位。
3. 对象选中后的联动描边。
4. 对话控制弹窗的平滑展开。
5. proposal 状态变化的轻量过渡反馈。

## 9. 响应式与展示适配

## 9.1 主要适配目标

- 桌面端主工作台
- 答辩大屏展示
- 中等尺寸笔记本演示

## 9.2 适配原则

1. 优先保证桌面端和大屏体验。
2. 总览页中央数字孪生画布应始终保持主导地位。
3. 智能体对话控制弹窗在中等宽度下可转为全屏覆盖式。
4. 移动端不是本期重点，但至少应保证核心信息不完全失效。

## 10. 前端实现建议

## 10.1 重构原则

建议遵循以下原则：

1. 尽量复用已有数据接口和状态编排。
2. 允许重做页面骨架和组件层次。
3. 允许新增独立的“数字孪生主屏组件”和“智能体对话控制组件”。
4. 避免继续堆叠通用卡片式后台布局。

## 10.2 建议新增或重构的核心组件

- `DigitalTwinOverviewScreen`
- `CityTwinCanvas`
- `FocusObjectRibbon`
- `DecisionRail`
- `RiskSignalBand`
- `AgentCommandDialog`
- `AgentControlConsole`
- `ImpactLinkDrawer`
- `ScenarioCompareBoard`

## 10.3 与现有代码关系

建议保留并改造以下入口：

- `App.tsx` 继续作为全局编排入口
- `OverviewPage.tsx` 重构为总览大屏
- `OperationsPage.tsx` 重构为行动处置编排台
- `AgentsPage.tsx` 重构为智能体协同透明页
- `GlobalRegionalProposalDialog.tsx` 可保留审批逻辑，但界面建议升级

## 11. 验收标准

完成 UI/UX 升级后，应至少满足以下标准：

1. 用户进入首页时，首先看到的是“数字孪生智能体洪水预警主屏”，而不是普通卡片后台。
2. 用户能够在主屏中完成重点对象选择、态势查看和风险理解。
3. 用户能够通过对话框直接控制智能体，而不必依赖页面跳转。
4. 主屏、对话框、审批和行动处置之间可以形成连续交互链路。
5. 页面风格能明确体现“数字孪生智能体洪水预警系统前端”的产品定位。
