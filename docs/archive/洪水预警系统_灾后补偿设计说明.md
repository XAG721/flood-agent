# 洪水预警系统灾后补偿设计说明

## 1. 设计定位
在当前系统中，灾后补偿不再被视为恢复或复盘阶段中的附属动作，而是被单独设计为 `Compensation` 阶段。其目标不是自动完成补偿决定，而是为补偿审核人员提供结构化的初筛结果、例外审查意见和人工复核队列。

因此，本阶段的系统定位是：
- 提供灾后补偿辅助审核
- 汇总多源补偿输入
- 识别规则筛查与个案合理性之间的冲突
- 输出可供人工审核的初步名单

## 2. 在主流程中的位置
系统状态机当前为：
- `Monitoring`
- `Attention`
- `Warning`
- `Response`
- `Rescue Coordination`
- `Recovery`
- `Compensation`
- `Postmortem`

补偿阶段位于 `Recovery` 之后、`Postmortem` 之前。这样可以将三类任务明确区分开：
- `Recovery`：恢复道路、安置点、基础服务和资源秩序
- `Compensation`：开展灾损筛查、补偿初审与人工复核队列生成
- `Postmortem`：沉淀经验和改进建议

## 3. 当前实现的数据来源
补偿阶段已不再只依赖观测和事件日志，而是直接读取专用补偿输入数据。当前原型从 `bootstrap_data/compensation_inputs` 目录加载以下三类 CSV：

### 3.1 住户申报数据
- `household_damage_claims.csv`

该表用于记录住户级申报信息，包括：
- 申报编号
- 家庭人数
- 老幼残情况
- 是否主住宅
- 是否地下空间进水
- 水深、估损、临时安置时长
- 已提交材料
- 当前申报状态

### 3.2 公共资产受损数据
- `public_asset_damage.csv`

该表用于记录公共资产受损情况，包括：
- 资产名称与类型
- 所属片区
- 损失等级
- 服务中断时长
- 估算损失
- 是否需要人工复核
- 责任单位

### 3.3 现场核查记录
- `field_verification_records.csv`

该表用于记录现场核查情况，包括：
- 核查对象编号
- 核查时间与人员
- 现场结论
- 优先级
- 审核建议
- 后续动作

## 4. 补偿阶段的知识与记忆支持
### 4.1 RAG 支持
补偿阶段会调用三类 RAG：
- `预案规则 RAG`：检索补偿政策、资格审查标准、人工复核规则
- `案例经验 RAG`：检索历史补偿争议、困难家庭审核经验、异常案例处理方式
- `区域画像 RAG`：检索区域内住户、重点设施、脆弱群体和受灾空间信息

### 4.2 长期记忆支持
补偿阶段会优先检索以下长期记忆：
- `compensation` 记忆
- `reflection` 记忆
- 必要时参考 `resource` 记忆

这使得补偿筛查不再是一次性判断，而是能够利用过往事件中的审查经验、争议模式和复核重点。

## 5. 补偿阶段的 Agent 结构
当前系统在补偿阶段采用“三步法”协同。

### 5.1 CompensationAgent
该 Agent 负责第一轮规则化筛查，主要任务包括：
- 汇总住户申报、公共资产受损和核查记录
- 结合规则文档生成初步筛查结果
- 输出规则意义上的初步通过、人工复核和暂缓名单
- 形成机器可读的结构化补偿载体

当前结构化输出中已包含：
- `proposed_approve_ids`
- `proposed_manual_review_ids`
- `proposed_defer_ids`
- `rule_based_reasons`

### 5.2 CompensationExceptionReviewAgent
该 Agent 负责第二轮例外审查，重点处理规则刚性与个案合理性之间的矛盾。其关注对象包括：
- 困难家庭
- 主住宅受灾但材料不全的案件
- 核查“基本属实”或“部分属实”的案件
- 申报与核查之间存在矛盾的案件

当前结构化输出中已包含：
- `exception_candidate_ids`
- `hardship_flags`
- `evidence_conflicts`
- `exception_reasons`

### 5.3 CompensationConflictResolverAgent
该 Agent 负责第三步收敛，不直接决定最终发放，而是基于前两轮结果给出：
- 初步通过队列
- 人工复核队列
- 暂缓队列
- 是否需要升级处理

当前结构化输出中已包含：
- `accepted_positions`
- `rejected_positions`
- `final_priority_order`
- `final_actions`
- `escalation_needed`
- `proposed_approve_ids`
- `proposed_manual_review_ids`
- `proposed_defer_ids`

## 6. 冲突类型与协商逻辑
补偿阶段的核心冲突是：

**补偿规则刚性 vs 个案合理性冲突**

该冲突的典型场景包括：
- 住户材料不齐，但现场核查基本属实
- 不完全满足书面条件，但属于脆弱家庭或主住宅高损失案件
- 群众申报和现场核查之间存在冲突
- 规则筛查倾向暂缓，但个案具有明显救助合理性

系统的协商流程为：
1. `CompensationAgent` 给出初步规则筛查结果
2. `CompensationExceptionReviewAgent` 标注例外候选与证据冲突
3. `CompensationConflictResolverAgent` 输出三类审核队列

## 7. 冲突触发规则
补偿冲突不是由 LLM 自主判断，而是先由确定性规则触发。当前触发条件包括：
- 存在例外候选案件
- 存在核查与申报冲突
- 存在主住宅或脆弱家庭高影响案件但规则结果不匹配

若未触发冲突，系统也会生成一张已解决的冲突卡片，用于说明“无实质冲突”。

## 8. 输出形式
### 8.1 文本建议
补偿阶段会生成：
- 模型结果摘要
- 审核建议
- 建议动作

### 8.2 结构化结果
补偿阶段会生成机器可读的队列和理由字段，便于后续系统展示、实验评估和人工审核。

### 8.3 冲突卡片
每次补偿协商都会生成 `ConflictCard`，记录：
- 触发原因
- 参与方
- 轮次摘要
- 最终收敛结论
- 是否需要人工复核
- 当前状态

这使得补偿阶段不是黑盒输出，而是可追踪、可解释的辅助审核链路。

### 8.4 对话式查询与确认执行
当前版本中，补偿模块已经接入后端驱动的 Copilot 会话层。值班员不再只能依赖固定按钮，而可以直接通过对话进行：
- 查询当前补偿审核状态
- 查看补偿审核方案摘要
- 查看人工复核队列和暂缓队列
- 追问已审核过的补偿相关方案历史

在执行方式上，补偿阶段遵循“先给方案，再确认执行”的原则：
- Copilot 先生成 `compensation_review` 方案记录
- 前端展示确认卡和审核边界说明
- 用户确认后，系统才真正执行补偿审核链路并刷新队列结果

这保证了补偿模块既具备对话式交互体验，又保持人工审核控制。

## 9. 日志与记忆写回
补偿阶段会写入：
- `compensation_recommendation`
- `conflict_detected`
- `negotiation_round_started`
- `negotiation_round_result`
- `conflict_resolved` 或 `conflict_escalated`
- `write_conflict_memory`

同时，补偿阶段的方案与会话也会在 SQLite 中留痕，主要包括：
- `plan_records` 中的 `compensation_review`
- Copilot 对话消息
- 补偿审核确认卡与执行结果

同时，补偿阶段的最终协商摘要会写入长期记忆库中的 `compensation` 类记忆，为后续类似事件提供可检索经验。

## 10. 人工审核边界
尽管系统已经能够读取补偿输入、识别冲突并生成审核队列，但它仍然不直接完成以下动作：
- 最终补偿资格裁定
- 最终审批签发
- 实际补偿金额发放

系统当前定位始终是“辅助审核”，而不是“自动发放”。这也是补偿模块在高风险业务场景下保持可控性的关键。

## 11. 当前版本结论
与早期设计相比，当前实现中的补偿阶段已经完成了三项关键升级：
- 从“只看观测和日志”升级为“直接读取住户申报、公共资产受损和核查记录”
- 从“单一补偿建议”升级为“规则筛查 + 例外审查 + 冲突仲裁”的协同机制
- 从“阶段性输出”升级为“结构化队列 + 冲突卡片 + 长期记忆写回”的完整闭环
- 从“固定流程入口”升级为“可通过对话查询、追问和确认执行”的智能体交互入口

因此，补偿模块已经从概念设计演进为可运行、可留痕、可演示、可用于论文分析的研究原型。
