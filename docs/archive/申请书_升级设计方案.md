# 申请书：升级设计方案

## 一、升级目标

本系统的升级目标，是将早期的洪水预警原型演进为一个可用于“区域级风险监测、对象级影响评估、个性化响应建议、受控 agentic 协作”的平台。

结合当前代码实现，升级目标已经明确分成三层：

- 第一层：生产化基础能力
- 第二层：受控 agentic Copilot
- 第三层：single-process multi-agent 协作

这意味着本系统不再只是“展示洪水等级”的演示程序，而是面向真实值班场景的指挥辅助系统。

## 二、当前已落地的升级内容

### 2.1 后端从单体原型升级为 v2 平台

当前系统的正式实现已经统一收敛到 `v2` 路径下，核心包括：

- 事件接入
- 风险计算
- 暴露分析
- 决策生成
- Copilot 编排
- Multi-agent 监督与共享记忆

对应代码位置：

- [flood_system/system.py](/d:/graduation_project/flood_system/system.py)
- [flood_system/api.py](/d:/graduation_project/flood_system/api.py)
- [flood_system/v2/platform.py](/d:/graduation_project/flood_system/v2/platform.py)

### 2.2 可更新数据源已经落地

系统已经不再依赖“启动时一次性加载”的静态数据，而是支持运行期动态更新。

已支持的数据源包括：

- 实体画像
- 区域资源基线
- 事件资源覆盖
- RAG 文档

对应能力：

- 画像 CRUD
- 事件级资源覆盖
- JSON 形式导入 RAG 文档
- RAG store 重载

### 2.3 Copilot 已升级为可解释 agentic 编排

Copilot 已经不是简单问答接口，而是采用：

- Rule planner
- LLM planner adapter
- Execution engine
- Reviewer
- Memory store

它可以输出：

- 为什么调用这些工具
- 为什么没调用另一些工具
- 每一步工具执行状态
- 数据新鲜度
- 证据缺口
- replan 次数
- completion status

### 2.4 Multi-Agent 第一阶段已实现

系统已经完成第一版 multi-agent 架构落地。

目前包含的 agent：

- HazardAgent
- ExposureAgent
- ResourceAgent
- PlanningAgent
- PolicyAgent
- CommsAgent

这些 agent 在单进程内通过：

- TaskBus
- AgentSupervisor
- EventSharedMemoryStore
- SupervisorLoopService

协同运行，并且可以通过 API 与前端界面观察执行状态。

### 2.5 后台定时运行机制已实现

系统已经实现真正的后台 supervisor loop，而不是只在人工触发时运行。

特点：

- FastAPI 启动时自动启动 loop
- 默认每 60 秒执行一次 background sweep
- 支持环境变量调整
- 支持 API 查询运行状态
- 前端 multi-agent 面板已展示 loop 运行状态、上次 sweep 和最近错误

## 三、当前系统的能力定位

从架构实现上看，当前系统可以定位为：

**面向洪水预警与对象级响应的受控 agentic AI 平台，具备 single-process multi-agent 协作能力。**

这个定位有三个关键词。

### 3.1 受控

高风险动作仍然保留人工审批边界，包括：

- proposal 批准
- 学校停课类动作
- 工厂停工类动作
- 养老院转移类动作
- 面向公众的大范围通知

### 3.2 Agentic

系统已经具备 agentic 的关键特征：

- 规划
- 工具调用
- review
- replan
- explain
- memory

### 3.3 Multi-Agent

系统已经具备多代理协作的第一版实现，但仍然是单进程版本，而不是分布式自治网络。

## 四、当前仍保留的边界

虽然系统已经显著升级，但从代码实现上看，仍然保留以下边界：

- 仍以 SQLite 为主存
- 尚未迁移到 PostgreSQL / Redis / PostGIS
- 尚未引入外部分布式任务队列
- multi-agent 仍是单进程 supervisor 版本
- 高风险动作仍由人工确认
- LLM planner 只做增强层，不直接接管最终决策

因此，本系统当前更适合被描述为：

**可用于真实值班辅助和演示验证的受控智能指挥平台**

而不是“完全无人干预的自主洪水大脑”。

## 五、升级方案的技术路线

结合已经落地的代码，后续升级可以按以下路线继续推进。

### 5.1 基础设施升级

- SQLite -> PostgreSQL
- 单机缓存 -> Redis
- 空间能力 -> PostGIS
- supervisor 单进程执行 -> 外部任务队列

### 5.2 平台治理升级

- 权限控制
- RBAC
- 审计日志
- 操作回放
- agent 评测集

### 5.3 多代理进一步升级

- 将 single-process multi-agent 演进为队列化 agent mesh
- 引入更强的 shared memory
- 增加 agent replay 与 supervisor dashboard

## 六、结论

根据当前代码，系统升级方案已经不再停留在设计层，而是已经落实为一个可运行的 v2 平台：

- 后端可运行
- 前端可操作
- RAG 可更新
- Copilot 可解释
- Multi-agent 可观测
- 后台 supervisor loop 可持续运行

因此，当前版本已经能够支撑“洪水预警与个性化响应平台”的核心展示与进一步扩展开发。
