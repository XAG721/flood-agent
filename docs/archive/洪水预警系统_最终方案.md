# 洪水预警系统最终方案

## 1. 系统定位
本系统面向区县级防汛应急指挥、街道应急管理和灾后核查等业务场景，目标是在山洪或城市内涝事件中提供从预警研判、响应调度到灾后补偿辅助审核和复盘沉淀的全过程支撑。当前实现为论文原型系统，强调流程闭环、职责清晰、实验可复现，而不追求生产级分布式部署。

系统整体采用“确定性核心引擎 + 状态机 + 工具调用型专家 Agent + 三类 RAG + 长期记忆”的架构。洪水预测、阈值判定、状态推进、审批和正式通知等关键环节保持确定性控制；智能体主要承担信息整合、风险研判、方案生成、冲突协商和经验沉淀等高认知负载任务。

## 2. 总体架构
系统由五个层次组成。

### 2.1 数据与模型层
数据侧接入监测观测、区域画像、资源状态、补偿输入和知识文档等信息。当前原型通过 CSV、DOCX 和 PDF 组织数据，后续可替换为数据库或外部平台接入。

模型侧采用 `DNO` 作为洪水预测算法，负责根据监测时序输出未来若干时段的风险趋势、峰值雨量和峰值水位等结果。`DNO` 只承担数值预测与趋势计算，不参与预警等级裁定和动作审批。

### 2.2 确定性核心引擎
确定性核心引擎负责：
- 调用 DNO 进行预测
- 根据阈值完成预警等级判定
- 推进业务状态机
- 创建审批任务
- 控制正式通知发送
- 记录审计日志

该部分是系统的安全边界。即使引入多智能体协同，风险等级、阶段转换、审批结果和正式动作仍由确定性逻辑控制。

### 2.3 智能体编排层
系统使用 `LangGraph` 管理主流程，使用 `LangChain` 的工具调用能力实现专家 Agent。当前实现不引入 LLM 总控 Agent，而是采用“确定性主流程 + 工具调用型专家 Agent”的方式，确保调度可控。

专家 Agent 在运行时只接收最小种子上下文，例如 `incident_id`、`area_id`、`stage` 和 `risk_level`，其余事实通过工具自行查询，包括：
- 最新预测结果
- 最近监测观测
- 区域画像
- 资源状态
- 补偿输入
- 历史事件日志
- 三类 RAG 文档
- 长期记忆
- 已生成的冲突卡片

### 2.4 知识与记忆层
知识层包含三类 RAG：
- `预案规则 RAG`：用于检索流程条款、预警模板、审批要求和补偿政策
- `案例经验 RAG`：用于检索历史洪水案例、处置经验和复盘结论
- `区域画像 RAG`：用于检索村镇、道路、桥梁、安置点、重点设施和脆弱人群信息

记忆层分为两类：
- 事件级上下文：通过 `Session`、事件日志和当前状态保存本次事件信息
- 长期记忆库：跨事件持久化存储可复用经验

当前长期记忆已细分为四类：
- `strategy`：预警或响应阶段形成的策略结论
- `resource`：资源调度和约束经验
- `compensation`：补偿筛查和人工复核经验
- `reflection`：复盘反思和通用教训

系统支持长期记忆检索与写回。后续相似事件发生时，Risk、Resource、Compensation 和 Postmortem 等 Agent 可按阶段、关键词和记忆类型优先级主动检索历史经验。

### 2.5 交互与执行层
系统当前通过 `FastAPI` 暴露两组接口，并由前端对话工作台统一访问。

第一组是传统事件流程接口，支持：
- `/incidents/ingest`
- `/incidents/ingest-from-csv`
- `/incidents/{incident_id}/process`
- `/incidents/{incident_id}/approve`
- `/incidents/{incident_id}/compensate`
- `/incidents/{incident_id}/finalize`
- `/incidents/{incident_id}`

其中 `/incidents/{incident_id}/approve` 已支持按 `plan_record_id` 对单个方案执行审批，而不再只处理整包动作。

第二组是对话式 Copilot 会话接口，支持：
- `/copilot/sessions/bootstrap`
- `/copilot/sessions/{session_id}`
- `/copilot/sessions/{session_id}/messages`
- `/copilot/sessions/{session_id}/proposals/{proposal_id}/confirm`
- `/copilot/sessions/{session_id}/proposals/{proposal_id}/cancel`

前端交互层已经改为“聊天主线程 + 结构化侧栏”的对话式工作台。值班员可以直接通过自然语言查询待审方案、已审方案、疏散规划、资源调度方案、通知留痕和补偿状态，也可以通过确认卡执行审批、补偿审核和复盘归档。

执行层目前采用模拟通知、模拟审批和本地 SQLite 持久化，适合论文演示与实验验证。当前持久化中除了事件状态外，还包括 `PlanRecord`、Copilot 会话、对话消息和确认卡片等数据。

## 3. 状态机设计
系统状态机如下：

- `Monitoring`
- `Attention`
- `Warning`
- `Response`
- `Rescue Coordination`
- `Recovery`
- `Compensation`
- `Postmortem`

其中：
- `Monitoring / Attention / Warning` 主要处理预警研判
- `Response / Rescue Coordination / Recovery` 主要处理应急响应与恢复
- `Compensation` 单独负责灾后补偿辅助审核
- `Postmortem` 负责复盘与经验沉淀

## 4. Agent 体系
### 4.1 核心专家 Agent
当前代码中已实现并接入工具调用的专家 Agent 包括：
- `ForecastAgent`
- `RiskAssessmentAgent`
- `CommunicationAgent`
- `EvacuationPlanningAgent`
- `ResourceCoordinationAgent`
- `CompensationAgent`
- `PostmortemAgent`

### 4.2 协商与仲裁 Agent
为处理复杂业务冲突，系统新增了四个协商相关 Agent：
- `FairnessReviewAgent`
- `CompensationExceptionReviewAgent`
- `ResponseConflictResolverAgent`
- `CompensationConflictResolverAgent`

这些 Agent 不独立控制主流程，而是在确定性触发条件满足时参与有限轮协商与收敛。

## 5. 三类冲突与协商机制
系统当前内置三类冲突协商场景。

### 5.1 风险优先级与资源约束冲突
该冲突关注高风险区域排序与实际可调度资源之间的矛盾。

协商流程为：
1. `RiskAssessmentAgent` 给出优先区域和最小资源需求
2. `ResourceCoordinationAgent` 根据当前资源状态指出资源缺口和无法覆盖区域
3. `ResponseConflictResolverAgent` 输出最终优先级和资源投放顺序

Warning 阶段会做资源预检，Response 阶段会再次正式协商。

### 5.2 疏散效率与脆弱人群公平性冲突
该冲突关注最快疏散方案与老幼残等脆弱群体覆盖之间的矛盾。

协商流程为：
1. `EvacuationPlanningAgent` 给出初始路线与安置点方案
2. `FairnessReviewAgent` 检查脆弱人群、路线可达性和安置点适配性
3. `ResponseConflictResolverAgent` 输出兼顾效率与公平性的最终方案

### 5.3 补偿规则刚性与个案合理性冲突
该冲突关注规则化筛查结果与个案困难、材料不完备但事实较充分等情况之间的矛盾。

协商流程为：
1. `CompensationAgent` 给出规则筛查的初步名单
2. `CompensationExceptionReviewAgent` 标识例外候选、困难个案和证据冲突
3. `CompensationConflictResolverAgent` 输出初步通过、人工复核和暂缓三类队列

补偿阶段仍保持“辅助审核”定位，不直接触发补偿发放。

## 6. 冲突卡片与机器可读载体
为了让协商过程可留痕、可展示、可验证，系统新增了两类结构化数据。

### 6.1 AgentRecommendation.structured_payload
各专家 Agent 在文本建议之外，还会输出机器可读的 `structured_payload`，用于冲突检测与后续协商。例如：
- Risk：`priority_zones`、`vulnerable_zones`、`minimum_resource_needs`
- Resource：`available_capacity`、`dispatchable_units`、`resource_gaps`、`unsupported_zones`
- Evacuation：`route_assignments`、`shelter_assignments`、`assisted_transfer_groups`
- Compensation：`proposed_approve_ids`、`proposed_manual_review_ids`、`proposed_defer_ids`

### 6.2 ConflictCard
系统新增 `ConflictCard` 结构，用于展示和存储协商结果，主要字段包括：
- `conflict_id`
- `incident_id`
- `stage`
- `conflict_type`
- `severity`
- `participants`
- `trigger_reason`
- `round_summaries`
- `final_resolution`
- `manual_review_required`
- `status`

每张冲突卡片都会写入数据库，并通过 `/incidents/{incident_id}` 返回给操作端。

### 6.3 PlanRecord 与细粒度审批
为了支持真正的对话式智能体查询与方案级审批，系统新增了 `PlanRecord` 作为统一方案载体。当前覆盖的方案类型包括：
- `warning_strategy`
- `evacuation_plan`
- `resource_plan`
- `communication_plan`
- `compensation_review`
- `postmortem_summary`

每条方案记录都带有标题、摘要、细节、动作包、来源 Agent、通知草稿、结构化载荷和审批人等字段。当前状态包括：
- `pending_approval`
- `approved`
- `executed`
- `cancelled`

在审批模型上，系统已从“单个总审批”演进为“方案级单独审批”：
- Warning 阶段先生成 `warning_strategy`，并单独审核
- 预警方案通过后进入 `Response`
- Response 阶段并行生成 `evacuation_plan`、`resource_plan`、`communication_plan`
- 这三类响应方案分别独立审核、独立留痕、独立执行

其中，通知发送、资源调度记忆写回和执行日志等副作用，都会在对应方案审批通过后再触发，而不再在预警方案通过时一次性整体执行。

## 7. 当前实现的流程闭环
### 7.1 预警研判
- 观测数据接入
- DNO 预测
- 阈值判断
- 风险等级确定
- Warning 阶段 Agent 协同
- 必要时进行资源预检协商
- 生成 `warning_strategy` 方案记录和对应审批任务
- 值班员可通过对话框查看待审方案、展开方案细节并确认审批

### 7.2 响应执行
- `warning_strategy` 审批通过后进入 `Response`
- 运行风险研判、资源协调、疏散规划、疏散公平审查和对外信息生成
- 将 `evacuation_plan`、`resource_plan`、`communication_plan` 分别落库为待审方案
- 值班员可通过对话框分别查询、审核和确认这三类方案
- 通知发送、资源调度记忆写回和执行留痕在对应方案通过后触发，仍走确定性链路

### 7.3 补偿阶段
- 进入 `Compensation` 阶段后读取住户申报、公共资产受损和现场核查记录
- 先进行规则筛查，再进行个案例外审查和仲裁收敛
- 输出初步通过、人工复核和暂缓队列
- 将补偿相关经验写入长期记忆库
- 补偿审核建议也会写入 `compensation_review` 方案记录，并通过对话确认卡触发执行

### 7.4 复盘阶段
- 进入 `Postmortem`
- 汇总事件日志、协商结果、长期记忆和案例知识
- 生成复盘建议并写入 `reflection` 记忆
- 复盘建议会以 `postmortem_summary` 方案记录形式保存，并通过对话确认执行

### 7.5 对话式 Copilot 工作流
在当前版本中，前端已不再依赖固定按钮驱动，而是以后端 Copilot 会话为中心。其流程为：
1. 恢复或创建 Copilot 会话
2. 读取当前事件、方案状态和对话上下文
3. 由模型优先完成意图识别，必要时回退到规则解析
4. 查询型请求直接回答，例如待审方案、已审方案历史、资源调度方案或通知留痕
5. 执行型请求先生成确认卡，再由用户确认执行

当前已经支持按方案种类筛选已审历史，例如：
- 查看已审核的通知方案
- 查看已审核的资源调度方案
- 查看已审核的疏散规划

## 8. 当前数据组织方式
原型系统目前支持：
- 监测观测数据：CSV 导入
- 区域画像与资源状态：CSV 读取
- 补偿输入：CSV 读取
- RAG 原始文档：从 `DOCX/PDF + meta.json` 自动扫描
- 运行时数据：SQLite 持久化

其中碑林区示例数据已经中文化，并覆盖了：
- 观测数据
- 区域画像
- 资源状态
- 补偿输入
- 三类 RAG 文档

## 9. 系统边界
本系统当前强调“智能体增强”，而不是完全自治。以下环节不交给 Agent 自主决定：
- DNO 数值预测本身
- 阈值判断
- 风险等级裁定
- 状态推进
- 审批结果
- 正式预警发布
- 最终补偿发放

Agent 的职责是：
- 检索知识
- 整合信息
- 形成建议
- 识别冲突
- 收敛方案
- 留存经验

## 10. 当前版本特点
当前版本已经具备以下特征：
- 专家 Agent 可自主调用工具
- 不同阶段支持不同 Agent 组合
- 具备跨事件长期记忆检索与写回机制
- 支持三类冲突的有限轮协商
- 支持灾后补偿辅助审核
- 支持完整的事件日志和冲突卡片留痕
- 支持后端驱动的对话式 Copilot 会话
- 支持方案级单独审批，而不是单个总审批
- 支持按方案种类查询待审、已审和最新方案状态
- 前端对话区支持即时进度提示和确认卡执行

因此，这一版本已经从“规则主导 + LLM 辅助生成”演进为“确定性主流程 + 工具调用型多 Agent + 冲突协商 + 长期记忆增强”的完整研究原型。
